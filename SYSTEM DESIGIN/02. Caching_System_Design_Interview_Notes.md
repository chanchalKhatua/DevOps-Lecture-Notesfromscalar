## 1. What is Caching & Why It Matters

> **Cache** = Temporary, small, fast storage that sits between the client and the actual data source (DB / file storage), storing frequently accessed or expensive-to-compute data.

### Two root causes that make caching necessary

| Trigger | Explanation | Example |
|---|---|---|
| Heavy computation | Result of expensive operation shouldn't be repeated | Analytics aggregation, multi-table JOINs |
| Heavy data for processing | Data needed per request is large, fetching every time is costly | Code judge test cases (~1 GB per problem) |

### Mental model
```
Request → [Cache] → hit? return fast
                 → miss? → DB → populate cache → return
```
- **Cache Hit** — data found in cache; returned immediately (nanoseconds).
- **Cache Miss** — data not in cache; fetched from DB, stored in cache for future requests.

**Cache is NOT the source of truth. The DB is.**

## 2. 5-Step Framework to Design a Cache

Use this framework in every system design interview where caching is relevant.

### Step 1 — Establish the Need for Caching

Ask: Is caching even required?
- Is the computation heavy (joins, analytics, ranking)?  
- Is the data size large per request (files, test cases)?  
- Is the same data read repeatedly by many users?

If yes → proceed with caching.

---

### Step 2 — Determine the Type of Cache

#### 2a. Local vs Global

| | Local Cache | Global Cache |
|---|---|---|
| **Where stored** | Inside the app server's own RAM / disk | Separate dedicated cache servers (Redis, Memcached) |
| **When to prefer** | Data is large (n/w transfer would negate gains); duplication across servers is acceptable | Data is small; avoiding duplication is important; multiple app servers need same data |
| **Tools** | Guava Cache, Caffeine Cache, Spring Boot Cache, in-memory HashMap | Redis, Memcached, AWS DynamoDB DAX |
| **Pros** | Extremely fast (no n/w hop), nanosecond RAM access | Consistent across all servers, no duplication |
| **Cons** | Cache inconsistency between servers, data duplication | Extra n/w hop adds latency, higher cost |

> **Rule of thumb:** Large file data (e.g., 1 GB test case files) → Local cache. Small metadata (e.g., rank list) → Global cache.

#### 2b. Single vs Distributed Cache *(only if you chose Global)*

| | Single Global Cache | Distributed Cache |
|---|---|---|
| **When** | Data to be cached is small; requests < ~50,000 req/sec | Data is very large OR traffic is very high |
| **How distributed?** | — | Multiple Redis instances, use consistent hashing / round robin for load balancing |

```
Local Cache:          Global - Single:          Global - Distributed:
  S1 [cache]            S1 ─┐                    S1 ─┐
  S2 [cache]            S2 ──→ [Redis]            S2 ──→ [R1]
  S3 [cache]            S3 ─┘                    S3 ──→ [R2]
```

---

### Step 3 — Determine the Invalidation Algorithm

Decide what consistency level is acceptable.

#### Consistency Models

| Model | Meaning | When to use |
|---|---|---|
| **Eventual Consistency** | Stale reads are OK; cache will sync within some Δt | Social feeds, analytics dashboards, rank lists |
| **Real-time (Strong) Consistency** | Cache must be as fresh as possible at all times | Code judge test case updates, banking, inventory |

#### Invalidation Strategies

**1. Cache Aside (Lazy Loading) + TTL** ← *most common, general-purpose*
- On read: check cache → hit: return; miss: go to DB, populate cache, return.
- Write goes directly to DB; cache is updated lazily (on next read or via TTL expiry).
- TTL controls max staleness.
- **Use when:** eventual consistency is acceptable; refilling cache is simple; data is small.

```
Read: Client → Server → Cache (HIT → return)
                              (MISS → DB → cache ← populate → return)
Write: Client → Server → DB (cache updated lazily via TTL)
```

**2. Write Through Cache** ← *real-time consistency*
- Write goes to Cache first, then synchronously to DB. Response sent only after both succeed.
- Cache is always in sync with DB.
- **Pros:** Reads always hit cache with fresh data.
- **Cons:** Increases write latency (two writes per request: cache + DB).
- **Use when:** real-time consistency required (e.g., code judge when test cases are updated by admin).

```
Write: Client → Server → Cache → (sync) → DB → success response
Read:  Client → Server → Cache (always fresh) → return
```

**3. Write Back Cache (Async Write Through)** ← *high write throughput*
- Write goes to Cache only. Response returned immediately. DB updated asynchronously in the background.
- **Pros:** Very low write latency; high write throughput.
- **Cons:** Risk of data loss if cache crashes before async DB write completes.
- **Use when:** you are OK with some data loss AND want high write throughput (e.g., analytics counters, gaming leaderboards with approximate scores).

> ⚠️ **Interview trap:** Only use write-back when data loss on cache failure is explicitly acceptable.

```
Write: Client → Server → Cache → success (immediate)
                              → DB (async, background)
```

**4. Write Around Cache**
- Writes bypass cache and go directly to DB. Cache is updated asynchronously later (DB pushes async call to cache).
- **Use when:** CDN updates; data is written rarely but read often.
- Best for **write-heavy workloads** where data is not immediately read.
    examples:
    - Bulk imports
    - Logging systems
    - Analytics pipelines
    - CDN updates (your existing example)

```
Write: Client → Server → DB → (async update) → Cache
Read: Client → Server → Cache (may be stale until async update arrives)
```

---

### Step 4 — Identify the Eviction Algorithm

When cache is full, which data to remove?

| Algorithm | Full Name | When evicted | Best use case |
|---|---|---|---|
| **LRU** | Least Recently Used | Item not accessed for the longest time | General purpose (most common in interviews) |
| **LFU** | Least Frequently Used | Item accessed fewest times overall | Rare/tough problems in code judge |
| **TTL** | Time To Live | After fixed time expires | Eventual consistency scenarios |
| **FIFO** | First In First Out | Oldest inserted item | Simple queues |
| **LIFO** | Last In First Out | Most recently inserted | Rarely used |
| **MRU** | Most Recently Used | Most recently accessed item | Specific workloads (e.g., rotating data) |

> **Interview default:** LRU unless there's a specific reason to choose otherwise.

---

### Step 5 — Load Balancing (if Distributed Cache)

| Need | Strategy |
|---|---|
| Replicas (multiple copies of same data) | Plain Round Robin |
| Sharding (different data on different nodes) | Consistent Hashing |

## 3. Types of Caches

### Caching Layers (from client to backend)

```
Browser → CDN → App Server (Local Cache) → Global Cache (Redis) → DB
```

| Layer | What's cached | Tools |
|---|---|---|
| **Browser** | DNS results, static assets (JS, CSS, images) | Built-in browser cache |
| **CDN** | Static files served from geo-nearest edge node | Akamai, Cloudflare, AWS CloudFront, Fastly |
| **Local (App Server)** | Large files, test cases, rarely changing data | Guava Cache, Caffeine, Spring Boot Cache |
| **Global (In-memory)** | Rank lists, session data, frequently queried derived data | Redis, Memcached, AWS DynamoDB DAX |

### Local Cache — Key Properties
- Storage lives inside the app server process (RAM or HDD of that server).
- **Extremely fast** — memory lookup in nanoseconds, no n/w hop.
- **Disadvantage:** Cache duplication across servers; risk of inconsistency between server instances.

### Global Cache — Key Properties
- Centralized cache servers shared by all app servers.
- Ensures consistency — no duplicates.
- **Disadvantage:** Extra n/w hop (~1ms) adds latency; higher infrastructure cost.

### Redis — Key Facts for Interviews
- Single-threaded key-value store.
- Supported value types: **String, Integer, List, Set, Sorted Set**.
- Commonly used for: rank lists, session storage, pub/sub, rate limiting.
- Can be configured as single instance or distributed cluster (Redis Cluster).

## 4. Cache Invalidation Strategies

> *"There are only two hard things in computer science: cache invalidation and naming things."* — Phil Karlton

### Summary Table

| Strategy | When cache is updated | Write latency | Read latency | Risk |
|---|---|---|---|---|
| Cache Aside + TTL | Lazily on read (after TTL expires) | Low (only DB) | Higher on miss | Stale data up to TTL |
| Write Through | Synchronously on every write | Higher (cache + DB) | Very low (always fresh) | Write bottleneck |
| Write Back | Asynchronously after write | Very low | Very low | Data loss on crash |
| Write Around | Asynchronously by DB push | Low (only DB) | Higher until async update | Temporary staleness |

## 5. Cache Eviction Algorithms

### Why eviction?
Cache has **limited capacity** (hardware RAM is expensive). When full, old entries must be removed to make room.

### LRU (Least Recently Used) — Interview Default
- Maintains order of access; evicts the entry that was accessed least recently.
- **Implementation:** HashMap + Doubly Linked List (O(1) get and put).
- **Great for:** General web traffic where recently accessed data is likely to be accessed again.

### LFU (Least Frequently Used)
- Tracks access frequency; evicts entry with lowest access count.
- **Great for:** Code judge (rare/hard problems accessed less frequently should be evicted first).

### TTL (Time To Live) — Not strictly eviction, but commonly used
- Each entry has a timestamp; automatically removed after TTL expires.
- **No data validity problem** if TTL is set appropriately.
- **No risk of stale data staying forever** in cache.

## 6. CDN — Content Delivery Network

### Problem CDN Solves
- Server in US, user in India → 10,000+ km → ~150ms round trip latency just from distance.
- Static assets (images, videos, JS) don't change per user → no need to fetch from origin every time.

### How CDN Works
```
Origin Server (US) → Pushes content to → CDN Edge Nodes (worldwide)
User (India) → DNS routes to nearest CDN node → CDN serves cached asset
```
- CDN node acts as a geo-distributed cache for static content.
- **Cache key:** CDN URL / content key → photo/video bytes.
- Updates use **Write Around**: DB/origin updates content → async push to CDN nodes.

### What is NOT true about CDNs (common MCQ trap)
- CDNs do NOT store health status of application servers ❌
- CDNs DO: reduce latency, serve geo-nearest content, use GeoDNS for routing ✓

## 7. Cache Challenges & Common Interview Pitfalls

### Challenge 1 — Cache Size is Limited
- Cache can only hold a fraction of total data.
- Solution: Smart eviction (LRU/LFU) + only cache hot/frequent data.

### Challenge 2 — Cache Stampede (Thundering Herd)
- Many requests simultaneously miss cache → all hit DB at once.
- Solutions: Mutex/lock on cache population; background refresh; probabilistic early expiration.

### Challenge 3 — Cache Inconsistency in Local Cache
- Two servers S1 and S2 each have a local cache for `cost(1)`.
- DB updates `cost(1)` to 25. S1's cache still shows 20 → **stale data**.
- Fix: Update cache whenever updating data in DB (use Write Through or TTL).

### Challenge 4 — Data Duplication in Local Cache
- Multiple servers caching the same data independently wastes RAM.
- Fix: Move to global cache if data size allows.

### Challenge 5 — Large File Local Cache
- If data is 1 GB per entry (e.g., test case files), global cache still transfers 1 GB over n/w on every request even after cache hit.
- Fix: **Local cache** — once downloaded to server, no further n/w transfer.

### Interview Traps to Avoid
| Wrong answer | Correct reasoning |
|---|---|
| "Use write-back for everything" | Write-back risks data loss; only use when explicitly acceptable |
| "Cache all data globally" | Large binary data belongs in local cache to avoid repeated n/w transfer |
| "LRU always best" | LFU better for skewed access patterns (rare vs. popular items) |
| "TTL solves all invalidation" | TTL still has a window of staleness; use Write Through for real-time needs |

## 8. Case Study 1 — Facebook News Feed

### Scale
- MAU: 3 Billion
- DAU: 2 Billion
- Average connections per user: 1,000

### DB Schema (Simplified)

```
Users:          id | name | gender | avatar_url | ...
User_Friends:   user_id_1 | user_id_2 | date_of_friendship
User_Posts:     post_id | author_id (user_id) | recipient_id (user_id) | content | ...
```

**Post size:** ~500 bytes
- post_id: 16 bytes (UUID)
- author_id: 16 bytes
- recipient_id: 16 bytes
- content: ~100 bytes
- attachment_url: ~200 bytes
- created_at / updated_at: 16 bytes each
- location, feeling: ~17 bytes

### Back-of-Envelope Calculation

**80-20-1 Rule:**
- 80% passive watchers
- 20% active watchers/engagers
- 1% content creators

```
Content creators = 1% of 2B DAU = 20M
Avg posts/creator/day = 5
Total posts/day = 5 × 20M = 100M posts/day

Over 20 years:
= 100M × 20 × 365
≈ 100M × 20 × 400    [approximation]
= 800 Billion posts

Storage = 800B × 500 bytes
= 800 × 10⁹ × 500
= 4 × 10¹¹ KB
= 4 × 10² TB
= 400 TB
```

### Sharding

- **Sharding key:** `user_id`
- All posts where a user is **author OR recipient** go to the same shard.
- Result: Multiple users per shard, but one user's data never spans shards.
- **Trade-off:** Post duplication across shards (a post from Vasu → Shubham appears in both Vasu's shard and Shubham's shard).

### News Feed Retrieval Problem

**Naive approach (Fan-out Read):**
```
When user opens newsfeed:
1. Fetch all friend_ids from DB
2. For each friend → go to their shard → fetch recent posts
3. Merge, rank, paginate → return
```
**Problem:** Friends' posts can be in ANY shard → queries fan out across all shards → **extremely expensive**.

### Ranking Factors for Newsfeed
1. Newness (recency)
2. Info/news value
3. Shock value
4. User-user affinity
5. Popularity index
6. Geo-location

> Facebook uses 2000+ parameters for ranking.

### Caching the Newsfeed

**Goal:** Pre-compute and cache top N posts per user.

**Design:**
```
1. DB read → fetch all friends' posts
2. Compute feed (rank by relevancy score)
3. Cache top 500 posts per user
```

**Cache size calculation (caching 500 posts × 50 pages for all 3B MAU users):**
```
500 bytes × 10 posts/page × 50 pages × 3B users
= 75 × 10,000 × 1B
= 750 TB  ← NOT feasible to cache
```

**Optimization — Cache only first 5 pages:**
```
500 bytes × 10 posts/page × 5 pages × 3B users
≈ 75 TB  ← More feasible (10× reduction)
```

**Problems with caching newsfeed:**

| Problem | Description |
|---|---|
| Size | Even 5 pages × 3B users is ~75 TB |
| Fan-out Write | Every new post by a user must invalidate/update caches of ALL their friends (up to 1000+ cache writes per post) |
| Algorithm change | Any change to ranking algorithm → invalidate everyone's cache |

**Cache pipeline:**
```
DB read (friends' posts) → Compute feed → Cache top 50 posts → Serve page 1-5
                                                               → Compute on-demand for page 6+
```

### Cache Type Decision for Facebook Newsfeed

- **Global cache** (Redis): stores derived newsfeed per user.
- **Eventual consistency** acceptable (small delay in seeing new posts is fine).
- **Invalidation:** TTL + Write-Through on post creation (fan-out write).

## 9. Case Study 2 — Scaler Code Judge

### Pillars of a Code Judge
1. Language selection
2. Problem statement / description
3. Code editor area
4. Autosave
5. Submit button
6. Evaluate code against all test cases

### APIs
```
GET  /problem/{id}       → fetch problem (Header: token)
POST /problem/{id}       → submit code  (Body: { code: "..." })
```

### DB Schema

```
Users:          id | name | email | password
User_Problems:  user_id | problem_id | score | status | code
Problems:       id | name | description | examples | score | time_limit | memory_limit | no_of_testcases
TestCases:      id | problem_id | input | output
```

### Test Case Storage

- Test case inputs/outputs are **large binary files**, not simple text rows.
- Input can be: int, array, String, Objects, HashMaps.
- **Storage:** S3 (Simple Storage Service) — bulk file storage, accessible via URL.
  ```
  www.s3.com/{account_id}/{filename}
  ```

**Size Calculation:**
```
N = 10⁶ problems
100 test cases/problem
8 bytes/integer

= 100 × 10⁶ × 8
= 10⁸ × 8 bytes
= 8 × 10⁵ KB
= 800 MB ≈ 1 GB (input file per problem)
+ 1 GB (output file per problem)
= 2 GB per problem

Total = 5,000 problems × 2 GB ≈ 5 TB
```

### Why Cache is Needed
- Test cases for a problem **rarely change**.
- Test cases are **fetched repeatedly** for every submission.
- Without cache: every submission triggers a 1 GB file transfer over network from S3.

### Local vs Global Cache Decision

| Criterion | Global Cache? | Local Cache? |
|---|---|---|
| Accessible to all app servers | ✅ Yes | ✅ Yes (each server downloads its own copy) |
| Large data makes sense | ✅ Yes for global | ✅ Yes for local too |
| N/w transfer overhead even after caching | ❌ Global still transfers 1 GB n/w per request | ✅ Local: once downloaded, no n/w transfer |
| Data duplication | ❌ (wasteful) | Minor concern; 100 GB per server is fine |

**Winner: Local Cache**

> Once a test case file is cached locally on an app server, all future submissions routed to that server get the file with zero n/w overhead.

### Cache Invalidation for Code Judge

**Problem:** When an admin updates test cases, only the server that processed the update will have fresh files. Other servers still serve old cached files.

**Solution: Versioning / Timestamps in filenames**
```
Old: p1_in.txt, p1_out.txt
New: p1_in_2024-03-16.txt, p1_out_2024-03-16.txt
```

**Flow:**
1. Admin submits new test cases.
2. New versioned files are uploaded to S3 (step 1).
3. DB is updated with new filenames (step 2).
4. When a server handles the next submission for this problem:
   - It reads filenames from DB → sees new filenames.
   - Checks local cache → miss (new filename not cached).
   - Downloads new file from S3 → caches it.
5. Old cached files on other servers are naturally invalidated because DB always returns new filenames.

**This is Write-Through for the admin path:**
```
Admin → LB → App Server (S1) → File Storage (S3) [upload new file with timestamp]
                             → DB [update filename record]
```

### Leaderboard Design

**Context:** 3-hour contest, 100,000 participants, 5 problems.

**Requirement:** Live leaderboard that updates with each submission.

**Problem with pure DB approach:**
- Every page load hits DB → sort 100,000 rows → very slow.

**Cache Solution:**
- Compute and cache rank list in **global cache (Redis Sorted Set)** every N seconds.
- `zadd leaderboard_<contest_id> <score> <user_id>`
- `zrange leaderboard_<contest_id> <start> <stop> WITHSCORES`

**Cache type:** Global cache (Redis) — all servers need same rank list, data is small.  
**Invalidation:** Write-through on every accepted submission → update Redis sorted set immediately.

## 10. Quick-Reference Cheat Sheet

### Choosing Cache Type
```
Is data large (>100MB per entry)?
  YES → Local Cache (avoid repeated n/w transfer)
  NO  → Global Cache (avoid duplication, ensure consistency)
    Is data volume very large OR traffic very high?
      YES → Distributed Cache (multiple Redis instances)
      NO  → Single Global Cache (one Redis)
```

### Choosing Invalidation Strategy
```
Is real-time consistency required?
  YES → Write Through Cache
  NO  (eventual consistency OK)?
    Is write heavy AND some data loss acceptable?
      YES → Write Back Cache
      NO  → Cache Aside + TTL
    Is data updated rarely but read often?
      YES → Write Around Cache (CDN updates, test cases)
```

### Choosing Eviction Algorithm
```
General use case → LRU
Access frequency matters (popular vs rare) → LFU
Time-based staleness is the concern → TTL
Simple queue-style cache → FIFO
```

### When to Use Write-Back — The Interview Litmus Test
Only use write-back if **ALL** of the following are true:
- [ ] Some data loss on crash is explicitly acceptable
- [ ] Write latency must be minimized
- [ ] Eventual DB sync is guaranteed (background worker)

### Cache Design Checklist for Any System
1. Does this system need caching? (heavy compute / large data / repeated reads)
2. Local or global? (data size, duplication tolerance)
3. If global: single or distributed? (data volume, request rate)
4. What consistency model? (eventual vs. real-time)
5. Which invalidation strategy? (Cache Aside / Write Through / Write Back / Write Around)
6. Which eviction algorithm? (LRU / LFU / TTL / FIFO)
7. If distributed: sharding (consistent hashing) or replicas (round robin)?
