## Jobs and CronJobs

### Kubernetes Jobs
Jobs are designed for **run-to-completion** workloads. Unlike Deployments that keep pods running continuously, Jobs ensure that a specified number of pods successfully complete their task.

#### Key Use Cases
- **Backup Tasks**: Periodic database backups (MySQL, PostgreSQL)
- **Batch Processing**: Generating monthly reports, analytics jobs
- **Cleanup Jobs**: Deleting old logs, cache, expired records
- **Data Import/Export**: ETL operations, data migrations
- **Security Scans**: Vulnerability scanning, compliance checks

#### Key Parameters

| Parameter | Description | Default |
|-----------|-------------|---------|
| `completions` | Number of successful pod completions required | 1 |
| `parallelism` | Number of pods to run simultaneously | 1 |
| `backoffLimit` | Number of retries before marking job as failed | 6 |
| `activeDeadlineSeconds` | Maximum time job can run before being terminated | None |

#### Job Execution Flow
```
Job → Pod 1 (exit 0) ✓
   → Pod 2 (exit 0) ✓
   → Pod 3 (exit 1) ✗ → Retry
   → Pod 3 (exit 0) ✓
   → Completed

If backoffLimit = 4 and container fails 4 times → Job Failed
```

#### Example: Job with Completions and Parallelism
```yaml
apiVersion: batch/v1
kind: Job
metadata:
  name: demo-job
spec:
  completions: 5        # Total 5 successful completions required
  parallelism: 2        # Run 2 pods at a time
  backoffLimit: 3       # Retry failed pods 3 times
  activeDeadlineSeconds: 60  # Job must finish within 60 seconds
  template:
    spec:
      containers:
      - name: worker
        image: busybox
        command: ["sh", "-c", "echo Running task; sleep 10; exit 0"]
      restartPolicy: Never
```

**Execution Pattern** (completions: 5, parallelism: 2):
- 2 pods start → 2 pods complete
- 2 pods start → 2 pods complete  
- 1 pod starts → 1 pod completes
- Total: 5 successful completions ✓

### CronJobs
CronJobs create Jobs on a **time-based schedule** using cron syntax.

#### CronJob Example
```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: demo-cronjob
spec:
  schedule: "*/2 * * * *"  # Runs every 2 minutes
  jobTemplate:
    spec:
      completions: 5
      parallelism: 2
      backoffLimit: 3
      activeDeadlineSeconds: 60
      template:
        spec:
          containers:
          - name: worker
            image: busybox
            command: ["sh", "-c", "echo Running task; sleep 10"]
          restartPolicy: Never
```

#### Suspend a CronJob
```bash
kubectl patch cronjob demo-cronjob -p '{"spec": {"suspend": true}}'
```

## Services

A **Service** provides **stable networking** for a group of Pods. It acts as a load balancer and provides a consistent endpoint even as Pods are created/destroyed.

**Analogy**: 
- Office Building → Kubernetes Cluster
- Multiple Employees → Pods
- Desk Phone → Pod IP (changes when employee moves)
- Helpline Number → Service (stable, routes to correct employee)

### Service Types

#### 1. ClusterIP (Default)
- **Used for**: Internal cluster communication only
- **Accessible**: Only within the cluster
- **Use Case**: Database services, internal APIs

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  type: ClusterIP
  ports:
  - protocol: TCP
    port: 80          # Service port
    targetPort: 80    # Container port
```

**How it works**:
```
Pod labels:        Service selector:
  app: nginx   →     app: nginx
                     ↓
                  Service routes to Pod
```

#### 2. NodePort
- **Used for**: Exposing services externally on each node
- **Port Range**: 30000-32767
- **Accessible**: `http://<Node_IP>:<NodePort>`

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  type: NodePort
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
    nodePort: 30007   # External access port
```

**Traffic Flow**:
```
curl http://NodeIP:30007
  ↓
Reaches nodePort on Node
  ↓
Routes to port 80 of Service
  ↓
Forwards to targetPort 80 on Container
```

**Drawbacks**:
1. Limited port range (30000-32767)
2. Requires access to node IPs
3. Security concerns
4. Not proper load balancing across nodes

#### 3. LoadBalancer
- **Used for**: Cloud environments (AWS, GCP, Azure)
- **Automatically creates**: Cloud provider's load balancer
- **Best for**: Production external access

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-nlb-service
  annotations:
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    service.beta.kubernetes.io/aws-load-balancer-scheme: "internet-facing"
    service.beta.kubernetes.io/aws-load-balancer-nlb-target-type: "ip"
spec:
  type: LoadBalancer
  ports:
  - port: 80
    targetPort: 80
    protocol: TCP
  selector:
    app: my-app
```

**How NLB Provisioning Works in EKS**:
1. Service of type `LoadBalancer` is created
2. Cloud Controller Manager (CCM) detects it
3. CCM communicates with AWS APIs
4. AWS creates Network Load Balancer (NLB)
5. Static Elastic IP is assigned
6. NLB forwards traffic to worker nodes → pods

#### 4. ExternalName
- **Used for**: Creating DNS alias to external services
- **No ClusterIP**: Just a CNAME record
- **Use Case**: Accessing external databases, legacy systems

```yaml
apiVersion: v1
kind: Service
metadata:
  name: external-db
spec:
  type: ExternalName
  externalName: db.xyz.com  # External hostname
```

**Multiple Deployments Use Case**:
```
Scenario: Multiple microservices using same external DB

5 Deployments → external-db service → db.xyz.com (RDS)

Benefit: If DB endpoint changes (db.xyz.com → db2.xyz.com),
         update only the Service, not all deployments
```

#### 5. Headless Service
- **No ClusterIP**: `clusterIP: None`
- **Returns**: Pod IPs directly (no load balancing)
- **Use Case**: Stateful applications (databases, Kafka)

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-headless
spec:
  clusterIP: None  # Makes it headless
  selector:
    app: nginx
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
```

**DNS Resolution**:
```bash
nslookup nginx-headless
# Returns: List of Pod IPs
10.244.1.5
10.244.2.8
10.244.3.12
```

## Ingress

**Ingress** is an API object that manages external HTTP/HTTPS access to services. It provides:
- **Host-based routing**: Different domains → different services
- **Path-based routing**: Different URL paths → different services
- **SSL/TLS termination**
- **Load balancing**

### Ingress vs NodePort/LoadBalancer

| Feature | NodePort | LoadBalancer | Ingress |
|---------|----------|--------------|---------|
| L7 Routing | ❌ | ❌ | ✅ |
| Path-based | ❌ | ❌ | ✅ |
| Host-based | ❌ | ❌ | ✅ |
| SSL/TLS | ❌ | ❌ | ✅ |
| Single IP | ❌ | 1 per service | ✅ Multiple services |

### Ingress Components
1. **Ingress Resource**: Rules (YAML manifest)
2. **Ingress Controller**: Implementation (NGINX, Traefik, HAProxy)

**Analogy**:
- Ingress Resource = Blueprint/Set of rules
- Ingress Controller = Worker that implements the rules

### Ingress Routing Types

#### 1. Host-Based Routing
```
app.example.com  → Service 1
api.example.com  → Service 2
```

#### 2. Path-Based Routing
```
example.com/app1  → Service 1
example.com/app2  → Service 2
```

### Demo Setup

**Prerequisites**:
1. Create Kind cluster
2. Install NGINX Ingress Controller
3. Create deployments and services
4. Create Ingress resource
5. Test

#### Install NGINX Ingress Controller
```bash
kubectl apply -f https://raw.githubusercontent.com/kubernetes/ingress-nginx/main/deploy/static/provider/kind/deploy.yaml

kubectl get pods -n ingress-nginx
```

#### Create Deployments with ConfigMaps
```yaml
---
apiVersion: v1
kind: ConfigMap
metadata:
  name: app1-config
data:
  index.html: |
    <html>
      <head><title>App 1</title></head>
      <body style="background-color: lightblue;">
        <h1>Welcome to App 1!</h1>
      </body>
    </html>
---
apiVersion: apps/v1
kind: Deployment
metadata:
  name: app1
spec:
  replicas: 1
  selector:
    matchLabels:
      app: app1
  template:
    metadata:
      labels:
        app: app1
    spec:
      containers:
      - name: nginx
        image: nginx
        ports:
        - containerPort: 80
        volumeMounts:
        - name: config-volume
          mountPath: /usr/share/nginx/html
      volumes:
      - name: config-volume
        configMap:
          name: app1-config
---
apiVersion: v1
kind: Service
metadata:
  name: app1-service
spec:
  selector:
    app: app1
  ports:
  - protocol: TCP
    port: 80
    targetPort: 80
```

#### Create Ingress Resource
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
  annotations:
    nginx.ingress.kubernetes.io/rewrite-target: /
spec:
  ingressClassName: nginx
  rules:
  - host: my-app.local
    http:
      paths:
      - path: /app1
        pathType: Prefix
        backend:
          service:
            name: app1-service
            port:
              number: 80
      - path: /app2
        pathType: Prefix
        backend:
          service:
            name: app2-service
            port:
              number: 80
```

#### Configure /etc/hosts
```bash
sudo vi /etc/hosts
# Add:
127.0.0.1 my-app.local
```

#### Test
```bash
curl http://my-app.local/app1
curl http://my-app.local/app2
```

### Traffic Flow
```
1. curl my-app.local/app1
2. /etc/hosts maps my-app.local → 127.0.0.1
3. Request reaches Ingress Controller (on control-plane:80)
4. Ingress Controller matches host + path in rules
5. Routes to app1-service
6. Service forwards to app1 pod
7. Response returns through same path
```

## Network Policies

**Network Policies** control traffic flow between pods, similar to firewall rules.

### Key Concepts
1. **podSelector**: Which pods the policy applies to
2. **policyTypes**: Ingress (incoming) / Egress (outgoing)
3. **Ingress rules**: Allowed incoming traffic sources
4. **Egress rules**: Allowed outgoing traffic destinations

### Example: Allow Frontend → Backend Communication
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-frontend-to-backend
  namespace: default
spec:
  podSelector:
    matchLabels:
      app: backend    # Apply to backend pods
  policyTypes:
  - Ingress
  ingress:
  - from:
    - podSelector:
        matchLabels:
          app: frontend  # Allow from frontend pods only
    ports:
    - protocol: TCP
      port: 80
```

**Effect**:
- ✅ Frontend pods can connect to Backend pods on port 80
- ❌ All other pods are blocked from Backend

### Container Network Interface (CNI)

| Feature | CNI (e.g., Calico, Cilium) | kube-proxy |
|---------|----------------------------|------------|
| **Primary Role** | Pod networking, IP allocation | Service networking, load balancing |
| **Scope** | Pod-to-Pod communication | Service-to-Pod routing |
| **How It Works** | Network interfaces, routes | iptables/ipvs rules |
| **Network Policies** | ✅ Supported (some CNIs) | ❌ Not supported |
| **Examples** | Flannel, Calico, Cilium, Weave | Built-in component |

### Demo: Network Policies with Calico

#### 1. Create Kind Cluster with Calico


## Storage and Volumes

### Container Storage Problem
- **Ephemeral Storage**: Data lost when:
  - Container restarts
  - Pod rescheduled
  - Pod deleted

**Solution**: Kubernetes Volumes

### Volume Types

#### 1. emptyDir
- **Lifetime**: Exists as long as Pod exists
- **Shared**: Between containers in same Pod
- **Use Cases**: 
  - Temporary cache
  - Sharing data between containers
  - Scratch space

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: myapp
spec:
  containers:
  - name: myapp
    image: alpine
    command: ['sh', '-c', 'while true; do echo "logging $(date)" >> /opt/logs.txt; sleep 1; done']
    volumeMounts:
    - name: data
      mountPath: /opt
  - name: logshipper
    image: alpine
    command: ['sh', '-c', 'tail -F /opt/logs.txt']
    volumeMounts:
    - name: data
      mountPath: /opt
  volumes:
  - name: data
    emptyDir: {}
```

**Comparison: Container Filesystem vs emptyDir**

| Feature | Container Filesystem | emptyDir Volume |
|---------|---------------------|-----------------|
| **Data Persistence** | Lost on container restart | Survives container restart |
| **Shared Storage** | No, isolated per container | Yes, shared across containers |
| **Performance** | Uses container's disk | Can be memory-backed |
| **Pod Deletion** | Data lost | Data lost |

**Memory-backed emptyDir** (for high-speed):
```yaml
volumes:
- name: cache
  emptyDir:
    medium: "Memory"  # Uses RAM instead of disk
```

#### 2. ConfigMap and Secrets as Volumes
- **Mounts**: Each key as a file
- **Dynamic Updates**: Content can be updated

```yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: app-config
data:
  key1: value1
  key2: value2
---
apiVersion: v1
kind: Pod
metadata:
  name: config-pod
spec:
  containers:
  - name: app
    image: nginx
    volumeMounts:
    - name: config
      mountPath: /tmp
  volumes:
  - name: config
    configMap:
      name: app-config
```

**Result** in `/tmp`:
```
/tmp/key1  (content: value1)
/tmp/key2  (content: value2)
```

#### 3. hostPath
- **Mounts**: Directory from Node's filesystem
- **Persistence**: Survives Pod deletion
- **Limitation**: Pod must be on same Node

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: hostpath-demo
spec:
  containers:
  - name: busybox
    image: busybox
    command: ["/bin/sh", "-c"]
    args:
    - while true; do
        echo "$(date) - Log from pod" >> /var/log/app-logs/demo.log;
        sleep 5;
      done
    volumeMounts:
    - mountPath: /var/log/app-logs
      name: host-volume
  volumes:
  - name: host-volume
    hostPath:
      path: /var/log/app-logs
      type: DirectoryOrCreate
```

**Limitations**:
1. **Not portable**: Pod tied to specific Node
2. **Security risk**: Pods can modify host files
3. **Scheduling complexity**: Pods must be on same Node

## Persistent Volumes (PV) and Claims (PVC)

### Why PV/PVC?

**Problem with hostPath**:
- Not portable across Nodes
- Manual management
- No dynamic provisioning

**Solution**: PV and PVC pattern

### Architecture
```
Warehouse (PV) → Storage capacity for specific inventory
     ↓
Request Storage (PVC) → Request specific space for data
     ↓
Pod uses PVC → Application accesses storage
```

### Persistent Volume (PV)
A **cluster-wide storage resource** provisioned by admin.

#### Key Components
1. **Capacity**: Storage size (e.g., 50Gi)
2. **Access Modes**: RWO, ROX, RWX
3. **Reclaim Policy**: Retain, Delete
4. **Storage Class**: Type of storage

### Access Modes

| Mode | Symbol | Description | Multiple Pods? | Multiple Nodes? |
|------|--------|-------------|----------------|-----------------|
| **ReadWriteOnce** | RWO | Single node, read-write | ❌ | ❌ |
| **ReadOnlyMany** | ROX | Multiple nodes, read-only | ✅ | ✅ |
| **ReadWriteMany** | RWX | Multiple nodes, read-write | ✅ | ✅ |

**Examples**:
- **RWO**: AWS EBS, GCE Persistent Disk (single instance databases)
- **ROX**: Shared configuration, static content
- **RWX**: NFS, EFS, CephFS (shared application data)

### Reclaim Policies

#### 1. Retain (Default)
```yaml
persistentVolumeReclaimPolicy: Retain
```
- PVC deleted → PV moves to "Released" state
- Data preserved
- Manual cleanup required

#### 2. Delete
```yaml
persistentVolumeReclaimPolicy: Delete
```
- PVC deleted → PV and underlying storage deleted
- Common with cloud storage (EBS, GCE PD)

### PV Example
```yaml
apiVersion: v1
kind: PersistentVolume
metadata:
  name: pv-retain
spec:
  capacity:
    storage: 5Gi
  accessModes:
  - ReadWriteOnce
  persistentVolumeReclaimPolicy: Retain
  storageClassName: standard
  hostPath:
    path: "/mnt/data-retain"
```

### Persistent Volume Claim (PVC)
A **request for storage** by a user/Pod.

```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: pvc-retain
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
  storageClassName: standard
```

### Binding Process
```
1. PVC created with requirements (5Gi, RWO, standard)
2. Kubernetes finds matching PV
3. PV bound to PVC (1:1 binding)
4. Pod can now use PVC
```

### Using PVC in Pod
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: pvc-test
spec:
  volumes:
  - name: storage
    persistentVolumeClaim:
      claimName: pvc-retain
  containers:
  - name: test-container
    image: busybox
    command: ["sleep", "3600"]
    volumeMounts:
    - mountPath: "/data"
      name: storage
```

### Comparison: RWO vs hostPath

| Feature | ReadWriteOnce (RWO) | hostPath |
|---------|---------------------|----------|
| **Data Location** | PV (EBS, Azure Disk) | Host node's filesystem |
| **Pod Scheduling** | K8s ensures pod on PV's node | Manual scheduling needed |
| **Security** | Managed storage | Pod can access host files |
| **Persistence** | Data persists even if node fails | Data lost if node replaced |

## Storage Classes

**Storage Class** enables **dynamic provisioning** of Persistent Volumes.

### Problem with Manual PV Provisioning
```
Scenario:
- Admin creates 50Gi PV
- User requests 10Gi PVC → Binds to 50Gi PV
- 40Gi wasted ❌

OR

- User requests 70Gi PVC
- No PV available → PVC stays Pending ❌
```

### Solution: Dynamic Provisioning
```
User creates PVC (20Gi) → Storage Class automatically provisions 20Gi PV
```

### Storage Class Components

#### 1. Provisioner
Defines which storage backend to use.

| Provisioner | Storage Backend |
|-------------|-----------------|
| `ebs.csi.aws.com` | AWS Elastic Block Store (EBS) |
| `pd.csi.storage.gke.io` | Google Persistent Disk |
| `disk.csi.azure.com` | Azure Disk |
| `nfs.csi.k8s.io` | NFS (Network File System) |
| `hostpath` | Host machine storage |
| `rancher.io/local-path` | Kind |

#### 2. Parameters
Backend-specific configurations.

```yaml
parameters:
  type: gp3          # AWS EBS volume type
  iops: "3000"       # Input/Output operations per second
  throughput: "125"  # MB/s
```

#### 3. Reclaim Policy
- `Retain`: Keep PV after PVC deletion
- `Delete`: Delete PV when PVC deleted

#### 4. Volume Binding Mode

**Immediate**:
```
PVC created → PV provisioned immediately (even if no Pod using it)
```

**WaitForFirstConsumer** (Recommended):
```
PVC created → PVC stays Pending
Pod created using PVC → PV provisioned on Pod's node
```

**Why WaitForFirstConsumer?**
- Ensures PV is created in same availability zone as Pod
- Avoids cross-AZ attachment issues

### Storage Class Example
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: fast-storage
provisioner: ebs.csi.aws.com
parameters:
  type: gp3
  iops: "3000"
  throughput: "125"
reclaimPolicy: Delete
volumeBindingMode: WaitForFirstConsumer
allowVolumeExpansion: true  # Enable PVC resize
```

### Using Storage Class
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: expandable-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 10Gi
  storageClassName: fast-storage  # References Storage Class
```

### Dynamic Provisioning Flow
```
1. User creates PVC (10Gi, fast-storage)
2. PVC stays Pending (WaitForFirstConsumer)
3. Pod created using PVC
4. Scheduler places Pod on Node-A
5. Storage Class provisions 10Gi PV on Node-A's zone
6. PV bound to PVC
7. Pod starts with storage
```

### Volume Expansion
```bash
# Increase PVC size from 10Gi to 20Gi
kubectl patch pvc expandable-pvc -p '{"spec":{"resources":{"requests":{"storage":"20Gi"}}}}'
```

### Volume Modes

#### Filesystem (Default)
```yaml
volumeMode: Filesystem
```
- PV formatted with filesystem (ext4, xfs)
- Pods access via file system interface
- Examples: NFS, EFS, CephFS

#### Block
```yaml
volumeMode: Block
```
- Raw block storage (no filesystem)
- Pods access as raw device (`/dev/xvdb`)
- Examples: AWS EBS, GCE Persistent Disk
- **Use Case**: Databases needing direct disk access

## Demo: Storage Class with Kind

### 1. Create Storage Class
```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: my-storage
provisioner: rancher.io/local-path  # Kind provisioner
reclaimPolicy: Retain
volumeBindingMode: WaitForFirstConsumer
```

### 2. Create PVC
```yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: my-pvc
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 2Gi
  storageClassName: my-storage
```

**Check PVC status**:
```bash
kubectl get pvc
# STATUS: Pending (waiting for Pod)
```

### 3. Create Pod
```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-pod
spec:
  volumes:
  - name: storage-volume
    persistentVolumeClaim:
      claimName: my-pvc
  containers:
  - name: app
    image: busybox
    command: ["sleep", "3600"]
    volumeMounts:
    - mountPath: /data
      name: storage-volume
```

**After Pod creation**:
```bash
kubectl get pvc
# STATUS: Bound ✓

kubectl get pv
# PV automatically created ✓
```

### 4. Test Data Persistence
```bash
# Write data
kubectl exec my-pod -- sh -c "echo 'Hello from Pod' > /data/test.txt"

# Delete Pod
kubectl delete pod my-pod

# Recreate Pod with same PVC
kubectl apply -f pod.yaml

# Verify data persists
kubectl exec my-pod -- cat /data/test.txt
# Output: Hello from Pod ✓
```

## Interview Questions & Answers

### 1. What happens if a Job's backoffLimit is reached?
The Job is marked as **Failed** and no more retries are attempted.

### 2. Difference between Deployment and Job?
- **Deployment**: Keeps Pods running continuously
- **Job**: Runs Pods until successful completion

### 3. When to use NodePort vs LoadBalancer vs Ingress?
- **NodePort**: Development/testing, small deployments
- **LoadBalancer**: Production cloud environments, simple exposure
- **Ingress**: Production, multiple services, path/host routing, SSL/TLS

### 4. Can multiple PVCs bind to the same PV?
**No**. PV to PVC binding is **1:1**.

### 5. What happens if PVC requests 10Gi but only 50Gi PV is available?
PVC binds to 50Gi PV, but **40Gi is wasted**. This is why Storage Classes are preferred.

### 6. Difference between emptyDir and PV?
- **emptyDir**: Pod-level, data lost when Pod deleted
- **PV**: Cluster-level, data persists across Pod lifecycle

### 7. Why use WaitForFirstConsumer?
Ensures PV is provisioned in the **same availability zone** as the Pod, avoiding cross-AZ attachment issues.

### 8. Can you resize a PVC?
**Yes**, if `allowVolumeExpansion: true` in Storage Class. Use:
```bash
kubectl patch pvc <name> -p '{"spec":{"resources":{"requests":{"storage":"<new-size>"}}}}'
```

### 9. Difference between Ingress and Service?
- **Service**: L4 (TCP/UDP) load balancing
- **Ingress**: L7 (HTTP/HTTPS) routing, path/host-based

### 10. How does Calico implement Network Policies?
Calico uses **iptables rules** and **BGP routing** to enforce network policies at the kernel level.

## Best Practices

### Jobs
✅ Set appropriate `activeDeadlineSeconds` to prevent runaway jobs  
✅ Use `backoffLimit` to control retry attempts  
✅ Monitor Job status with `kubectl get jobs`

### Services
✅ Use ClusterIP for internal services  
✅ Use Ingress instead of multiple LoadBalancers  
✅ Label Pods correctly for Service selectors

### Storage
✅ Use Storage Classes for dynamic provisioning  
✅ Set appropriate `reclaimPolicy` (Delete for dev, Retain for prod)  
✅ Use `WaitForFirstConsumer` for cloud environments  
✅ Enable `allowVolumeExpansion` for future growth  
✅ Use RWX for shared data, RWO for single-instance apps

### Network Policies
✅ Start with deny-all, then whitelist  
✅ Use namespace selectors for cross-namespace communication  
✅ Test policies in dev before production  
✅ Document network policy rules
