## StatefulSets

### Overview
A **StatefulSet** is a Kubernetes workload API object designed to manage **stateful applications** that require:
- Stable, unique network identities
- Stable, persistent storage
- Ordered, graceful deployment and scaling
- Ordered, automated rolling updates

### StatefulSet vs Deployment

| Feature | Deployment | StatefulSet |
|---------|-----------|-------------|
| **Use Case** | Stateless applications | Stateful applications (databases, etc.) |
| **Pod Identity** | Random names, no stable identity | Stable, unique identities (predictable names) |
| **Storage** | Ephemeral or shared storage | Persistent storage with PVCs |
| **Scaling** | Pods scaled randomly | Pods scaled sequentially (0→1→2) |
| **Service Type** | Normal Service (load balancing) | Headless Service (unique DNS per pod) |
| **Pod Naming** | `app-xyz-12345678` | `app-xyz-0`, `app-xyz-1`, `app-xyz-2` |

### Key Concepts

#### Pod Naming Convention
```
<statefulset-name>-<ordinal-index>
Example: my-db-0, my-db-1, my-db-2
```

#### Headless Service
- Created with `clusterIP: None`
- Provides unique DNS entries for each pod
- DNS format: `<pod-name>.<service-name>.<namespace>.svc.cluster.local`
- Example: `my-db-0.db-service.my-namespace.svc.cluster.local`

### Complete StatefulSet Example

```yaml
# Headless Service
apiVersion: v1
kind: Service
metadata:
  name: db-service
spec:
  clusterIP: None  # Makes service headless
  selector:
    app: my-db
  ports:
  - port: 5432
    targetPort: 5432

---
# StatefulSet
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: my-db
spec:
  serviceName: db-service  # Links to headless service
  replicas: 3
  selector:
    matchLabels:
      app: my-db
  template:
    metadata:
      labels:
        app: my-db
    spec:
      containers:
      - name: database
        image: postgres
        env:
        - name: POSTGRES_PASSWORD
          value: "mysecretpassword"
        ports:
        - containerPort: 5432
        volumeMounts:
        - name: db-storage
          mountPath: /var/lib/postgresql/data
  
  # Volume Claim Templates - creates separate PVC for each pod
  volumeClaimTemplates:
  - metadata:
      name: db-storage
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 5Gi
```

### Interview Questions
**Q: Why use StatefulSets instead of Deployments for databases?**
- Stable network identity (each pod has predictable DNS name)
- Persistent storage that survives pod rescheduling
- Ordered scaling ensures data consistency
- Individual pod addressing for master-slave configurations

**Q: What happens when a StatefulSet pod is deleted?**
- Pod is recreated with the **same name and identity**
- Reattaches to the **same PVC** (data persists)
- Maintains the same ordinal index

## Kubernetes Security Framework

### Security Layers
```
┌─────────────────────────────────────┐
│   1. Authentication (Who are you?)  │
├─────────────────────────────────────┤
│   2. Authorization (What can you do?)│
├─────────────────────────────────────┤
│   3. Admission Control (Policy checks)│
└─────────────────────────────────────┘
```

### API Request Flow
```
User/Service → Authentication → Authorization → Admission Controllers → API Server → etcd
```

## Authentication Methods

### Overview
**Authentication verifies the identity** of users and systems interacting with the Kubernetes cluster.

### 1. Client Certificates (X.509)

**How it works:**
- Uses X.509 certificates to authenticate users
- Certificate issued by Kubernetes CA
- Strong cryptographic validation

**Use Case:** Secure authentication for admins and system processes

**Pros:**
- Strong security
- Cryptographic validation

**Cons:**
- Requires certificate management
- Complex setup

**Workflow:**
```
1. Generate private key: openssl genrsa -out user.key 2048
2. Create CSR: openssl req -new -key user.key -out user.csr
3. Submit CSR to Kubernetes CA
4. CA signs and returns certificate
5. Use certificate in kubeconfig
```

### 2. Service Account Tokens (Bearer Token)

**How it works:**
- Kubernetes automatically generates tokens for pods
- Token mounted at `/var/run/secrets/kubernetes.io/serviceaccount/token`
- Pods use this token to authenticate with API server

**Use Case:** Applications running inside the cluster

**Token Characteristics:**
- Automatically mounted to pods
- Scoped to namespace
- Can be rotated for security

**Pros:**
- Automatic, easy to use
- Built into Kubernetes

**Cons:**
- Security risks if token leaked
- Requires token rotation strategy

**Example:**
```bash
# View token inside pod
kubectl exec -it <pod-name> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token
```

### 3. Static Token File

**How it works:**
- Predefined tokens stored in CSV file
- Format: `token,username,uid,group`
- Configured in API server manifest

**Use Case:** Testing or simple setups (NOT production)

**Configuration:**
```csv
# /etc/kubernetes/static_tokens.csv
my-static-token,user1,user1,system:masters
```

API Server configuration:
```yaml
--token-auth-file=/etc/kubernetes/static_tokens.csv
```

**Pros:**
- Easy setup

**Cons:**
- **INSECURE** - token never expires
- Requires API server restart to update
- Manual management

### 4. OpenID Connect (OIDC) Authentication

**How it works:**
- Integrates with external identity providers (Google, Azure, Okta)
- Uses OAuth 2.0/OIDC protocol
- Centralized identity management

**Flow:**
```
User → Identity Provider → Token → Kubernetes API Server
```

**Use Case:** Enterprise environments with centralized authentication

**Pros:**
- Centralized identity management
- Easy integration with corporate credentials
- Supports MFA

**Cons:**
- Requires external identity provider
- More complex setup

**Example Identity Providers:**
- Google
- Azure Active Directory
- Okta
- Keycloak

### 5. Webhook Token Authentication

**How it works:**
- API server calls external webhook to validate tokens
- Custom authentication logic
- Flexible authentication service

**Flow:**
```
User Request → API Server → Webhook Server → Validation Response → Access Decision
```

**Use Case:** Custom authentication systems

**Configuration:**
```yaml
# API Server flags
--authentication-token-webhook-config-file=/etc/kubernetes/webhook-config.yaml
--authentication-token-webhook-cache-ttl=5m
```

**Webhook Config:**
```yaml
apiVersion: v1
kind: Config
clusters:
- name: webhook-auth
  cluster:
    server: https://auth.example.com/validate
users:
- name: webhook-auth
contexts:
- context:
    cluster: webhook-auth
    user: webhook-auth
  name: webhook-auth
current-context: webhook-auth
```

**Pros:**
- Flexible, integrates with external systems
- Custom authentication logic

**Cons:**
- Requires maintaining separate authentication service
- Additional complexity

### Authentication Methods Comparison

| Method | Security | Complexity | Use Case | Recommended |
|--------|----------|----------|----------|-------------|
| **Client Certificates** | High | Medium | Admin access | ✅ Yes |
| **Service Account Tokens** | Medium | Low | In-cluster apps | ✅ Yes |
| **Static Token File** | Low | Low | Testing only | ❌ No (production) |
| **OIDC** | High | High | Enterprise SSO | ✅ Yes |
| **Webhook** | Medium-High | High | Custom systems | ⚠️ Case-by-case |

## Authorization Modes

### Overview
**Authorization determines what actions an authenticated user can perform** within the cluster.

### 1. Node Authorization

**Purpose:** Authorizes kubelets to read and write node-specific objects

**How it works:**
- Used by kubelet on each node
- Limited to node-specific operations
- Automatic, built-in

**Permissions:**
- Read: pods, services, endpoints
- Write: node status, pod status, events

### 2. ABAC (Attribute-Based Access Control)

**How it works:**
- Policies stored in static JSON file
- Each policy defines user, resource, and actions
- Requires API server restart for updates

**Policy Example:**
```json
{
  "apiVersion": "abac.authorization.kubernetes.io/v1beta1",
  "kind": "Policy",
  "spec": {
    "user": "alice",
    "namespace": "default",
    "resource": "pods",
    "readonly": true
  }
}
```

**API Server Configuration:**
```bash
--authorization-mode=Node,RBAC,ABAC
--authorization-policy-file=/etc/kubernetes/abac-policy.json
```

**Why ABAC is Less Preferred:**
- ❌ Static file-based (requires API server restart)
- ❌ Not scalable for large clusters
- ❌ Difficult to manage
- ✅ RBAC is preferred (dynamic, easier to manage)

### 3. Webhook Authorization

**How it works:**
- Delegates authorization decisions to external service
- API server sends request to webhook
- Webhook validates and returns allow/deny

**Flow:**
```
API Request → API Server → Webhook Server → Authorization Decision → Allow/Deny
```

**Configuration:**
```yaml
apiVersion: v1
kind: Config
clusters:
- name: webhook-auth
  cluster:
    server: https://auth.example.com/validate
users:
- name: webhook-auth
contexts:
- context:
    cluster: webhook-auth
    user: webhook-auth
  name: webhook-auth
current-context: webhook-auth
```

### 4. RBAC (Role-Based Access Control) ⭐ **RECOMMENDED**

#### Key Concepts

**Roles:** Define permissions (what can be done)
**RoleBindings:** Assign roles to users/groups (who can do it)

#### Components

1. **Role** - Namespace-scoped permissions
2. **ClusterRole** - Cluster-wide permissions
3. **RoleBinding** - Assigns Role to subjects within namespace
4. **ClusterRoleBinding** - Assigns ClusterRole to subjects cluster-wide

#### Role Example
```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]  # "" indicates core API group
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
```

#### RoleBinding Example
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: User
  name: "jane"
  apiGroup: rbac.authorization.k8s.io
- kind: Group
  name: "dev-team"
  apiGroup: rbac.authorization.k8s.io
- kind: ServiceAccount
  name: "app-sa"
  namespace: default
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

#### ClusterRole Example (Cluster-wide)
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-admin-reader
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]
```

### RBAC API Groups & Resources

| API Group | Example Resources |
|-----------|-------------------|
| `""` (core) | pods, services, configmaps, secrets, nodes, namespaces |
| `apps` | deployments, daemonsets, statefulsets, replicasets |
| `batch` | jobs, cronjobs |
| `rbac.authorization.k8s.io` | roles, rolebindings, clusterroles, clusterrolebindings |
| `networking.k8s.io` | networkpolicies, ingresses |
| `storage.k8s.io` | storageclasses, volumeattachments |

### RBAC Verbs (Actions)

| Verb | Description |
|------|-------------|
| `get` | Read a specific resource |
| `list` | List all resources of a type |
| `watch` | Monitor changes to resources in real-time |
| `create` | Create a new resource |
| `update` | Modify an existing resource |
| `patch` | Partially update an existing resource |
| `delete` | Remove a resource |
| `deletecollection` | Delete multiple resources at once |

### Authorization Demo Workflow

```
1. Generate key → openssl genrsa -out dev-user.key 2048
2. Create CSR → openssl req -new -key dev-user.key -out dev-user.csr
3. Submit CSR to CA → kubectl certificate approve
4. Create Role → kubectl create -f role.yaml
5. Create RoleBinding → kubectl create -f rolebinding.yaml
6. Configure kubeconfig with certificate and key
7. Test permissions → kubectl auth can-i get pods --as dev-user
```

## Security Contexts

### Overview
**Security contexts define privilege and access control settings** for pods and containers.

### Two Levels

1. **Pod-level:** Applies to all containers in the pod
2. **Container-level:** Applies to specific container (overrides pod-level)

### Pod-Level Security Context

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:  # Applies to ALL containers
    runAsUser: 1000      # Run as non-root user (UID 1000)
    runAsGroup: 3000     # Group ID
    fsGroup: 2000        # File system group ownership
  containers:
  - name: app-container
    image: nginx
```

### Container-Level Security Context

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-container
spec:
  containers:
  - name: app-container
    image: nginx
    securityContext:  # Only applies to THIS container
      runAsUser: 2000
      readOnlyRootFilesystem: true
      allowPrivilegeEscalation: false
```

### Key Security Context Fields

#### 1. runAsUser & runAsGroup

```yaml
securityContext:
  runAsUser: 1000   # Runs as non-root user (UID 1000)
  runAsGroup: 3000  # Uses GID 3000
```

**Interview Q:** Why run as non-root?
- Limits potential damage from container compromise
- Follows principle of least privilege
- Prevents privilege escalation

#### 2. Privileged Mode

```yaml
securityContext:
  privileged: true  # Grants FULL access to host
```

⚠️ **Dangerous:** Container has unrestricted access to host resources

#### 3. allowPrivilegeEscalation

```yaml
securityContext:
  allowPrivilegeEscalation: false  # Prevents gaining more privileges
```

**Recommended:** Always set to `false` unless absolutely necessary

#### 4. readOnlyRootFilesystem

```yaml
securityContext:
  readOnlyRootFilesystem: true  # Prevents modifying root filesystem
```

**Benefits:**
- Prevents malware installation
- Ensures immutable containers
- Improves security posture

#### 5. Linux Capabilities

```yaml
securityContext:
  capabilities:
    add: ["NET_ADMIN"]   # Allows network configuration
    drop: ["ALL"]        # Drops all unnecessary capabilities
```

**Common Capabilities:**
- `NET_ADMIN` - Network configuration
- `SYS_TIME` - Set system time
- `CHOWN` - Change file ownership

### Complete Security Context Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:  # Pod-level
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
  containers:
  - name: secure-container
    image: busybox
    securityContext:  # Container-level
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    command: ["sleep", "3600"]
```

### Testing Security Contexts

```bash
# Check user ID
kubectl exec -it secure-pod -- id

# Try to escalate to root (should fail)
kubectl exec -it secure-pod -- sh -c "su root"

# Try to create file in root (should fail with readOnlyRootFilesystem)
kubectl exec -it secure-pod -- touch /testfile
```

### Security Context Best Practices

✅ **DO:**
- Always run as non-root user
- Set `allowPrivilegeEscalation: false`
- Use `readOnlyRootFilesystem: true`
- Drop unnecessary capabilities
- Use `fsGroup` for shared volume access

❌ **DON'T:**
- Run containers as root (UID 0)
- Use `privileged: true` unless absolutely necessary
- Allow privilege escalation
- Grant unnecessary capabilities

## Admission Controllers

### Overview
**Admission controllers intercept API requests AFTER authentication and authorization** but BEFORE the object is persisted to etcd.

### Request Flow
```
API Request → Authentication → Authorization → Admission Controllers → etcd
                                                      ↓
                                            Validate/Modify/Reject
```

### Types of Admission Controllers

#### 1. Mutating Admission Controllers
- **Modify** objects before they are stored
- Examples: Add default labels, inject sidecars, set resource limits

#### 2. Validating Admission Controllers
- **Validate** objects against policies
- **Cannot modify** objects
- Examples: Enforce resource quotas, reject insecure configurations

#### 3. Dual-Purpose Controllers
- Both mutate AND validate
- Example: **LimitRanger**

### Execution Order
```
Mutating Controllers → Validating Controllers → Persist to etcd
```

### Common Admission Controllers

| Controller | Type | Purpose |
|-----------|------|---------|
| **MutatingAdmissionWebhook** | Mutating | Calls external webhook to modify requests |
| **ValidatingAdmissionWebhook** | Validating | Calls external webhook to validate requests |
| **AlwaysPullImages** | Mutating | Forces pulling images from registry |
| **DefaultStorageClass** | Mutating | Assigns default storage class to PVCs |
| **LimitRanger** | Both | Adds default resource limits, validates pod resource requests |
| **ResourceQuota** | Validating | Enforces namespace-level quotas |
| **NodeRestriction** | Validating | Restricts kubelet from modifying nodes/pods |
| **NamespaceLifecycle** | Validating | Prevents operations in non-existent/terminating namespaces |

### LimitRanger - Dual Purpose Example

**Mutating behavior:** Adds default resource requests/limits if missing
**Validating behavior:** Rejects pods that exceed limits

#### Enable LimitRanger

```bash
# Edit API server manifest
vi /etc/kubernetes/manifests/kube-apiserver.yaml

# Add to --enable-admission-plugins
--enable-admission-plugins=NodeRestriction,LimitRanger

# API server pod automatically restarts
```

#### LimitRange Resource

```yaml
apiVersion: v1
kind: LimitRange
metadata:
  name: pod-resource-limits
  namespace: limitranger-demo
spec:
  limits:
  - type: Container
    default:               # Default limits (if not specified)
      cpu: "500m"
      memory: "256Mi"
    defaultRequest:        # Default requests (if not specified)
      cpu: "250m"
      memory: "128Mi"
    max:                   # Maximum allowed
      cpu: "1"
      memory: "512Mi"
    min:                   # Minimum required
      cpu: "100m"
      memory: "64Mi"
```

### Webhook Admission Controllers

#### MutatingAdmissionWebhook
```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingWebhookConfiguration
metadata:
  name: example-mutating-webhook
webhooks:
- name: mutate.example.com
  clientConfig:
    service:
      name: webhook-service
      namespace: default
      path: "/mutate"
    caBundle: <base64-encoded-ca-cert>
  rules:
  - operations: ["CREATE"]
    apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
```

#### ValidatingAdmissionWebhook
```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: ValidatingWebhookConfiguration
metadata:
  name: example-validating-webhook
webhooks:
- name: validate.example.com
  clientConfig:
    service:
      name: webhook-service
      namespace: default
      path: "/validate"
    caBundle: <base64-encoded-ca-cert>
  rules:
  - operations: ["CREATE", "UPDATE"]
    apiGroups: [""]
    apiVersions: ["v1"]
    resources: ["pods"]
  failurePolicy: Fail  # Reject if webhook unavailable
```

### Interview Questions

**Q: What's the difference between Authorization and Admission Control?**
- **Authorization:** Determines if user CAN perform action (RBAC)
- **Admission Control:** Enforces policies and validates/modifies the request CONTENT

**Q: When would you use Webhook Admission Controllers?**
- Custom validation logic (e.g., check image signatures)
- Policy enforcement (e.g., reject pods without resource limits)
- Mutation logic (e.g., inject sidecar containers)
- Integration with external systems

**Q: What happens if a Validating Admission Webhook fails?**
- Depends on `failurePolicy`:
  - `Fail`: Request is rejected (safe default)
  - `Ignore`: Request proceeds (can cause security issues)

## Kubernetes API Versioning

### API Version Lifecycle

```
Alpha (v1alpha1) → Beta (v1beta1) → Stable (v1)
```

### Version Stages

| Stage | Characteristics | Production Use |
|-------|----------------|----------------|
| **Alpha** | Experimental, may change/be removed | ❌ No |
| **Beta** | More stable, guaranteed not to be removed, may have minor changes | ⚠️ Testing only |
| **Stable (GA)** | Fully tested, no breaking changes | ✅ Yes |

### API Version Format

```
<apiVersion>: <group>/<version>
```

**Examples:**
- `v1` - Core API (no group)
- `apps/v1` - Apps group
- `batch/v1` - Batch group
- `networking.k8s.io/v1` - Networking group

### Deprecation vs Removal

**Deprecated:** Still available, but not recommended (will be removed in future)
**Removed:** No longer compatible, must migrate to new version

### Evolution Example: Ingress

```
v1.1 (2015)  → networking.k8s.io/v1beta1  (Introduced)
v1.19 (2020) → networking.k8s.io/v1       (GA - General Availability)
v1.22 (2021) → v1beta1 removed            (Must use v1)
```

### API Groups

| API Group | Example Resources |
|-----------|-------------------|
| Core API (`""`) | pods, services, configmaps, secrets |
| `apps` | deployments, statefulsets, daemonsets |
| `batch` | jobs, cronjobs |
| `networking.k8s.io` | networkpolicies, ingresses |
| `storage.k8s.io` | storageclasses, volumeattachments |
| `certificates.k8s.io` | certificatesigningrequests (CSR) |
| `admissionregistration.k8s.io` | mutating/validating webhook configurations |

### Deployment API Evolution Example

```yaml
# Kubernetes v1.8
apiVersion: apps/v1beta1
kind: Deployment

# Kubernetes v1.9
apiVersion: apps/v1beta2
kind: Deployment

# Kubernetes v1.10+
apiVersion: apps/v1
kind: Deployment

# v1.15: apps/v1beta1 and apps/v1beta2 REMOVED
```

### Ingress Migration Example

**Old (v1beta1) - Removed in v1.22:**
```yaml
apiVersion: networking.k8s.io/v1beta1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: my-app.example.com
    http:
      paths:
      - path: /
        backend:
          serviceName: my-service  # Old format
          servicePort: 80          # Old format
```

**New (v1) - Required in v1.22+:**
```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: my-ingress
spec:
  rules:
  - host: my-app.example.com
    http:
      paths:
      - path: /
        pathType: Prefix         # Required field
        backend:
          service:               # New nested structure
            name: my-service
            port:
              number: 80
```

### Converting API Versions

```bash
# Convert old manifest to new version
kubectl convert -f old-manifest.yaml --output-version apps/v1

# Dry-run to check conversion
kubectl convert -f old-manifest.yaml --output-version apps/v1 --dry-run=client -o yaml
```

### Interview Questions

**Q: How do you check which API versions are supported in your cluster?**
```bash
kubectl api-versions
kubectl api-resources
```

**Q: What's the difference between deprecated and removed?**
- **Deprecated:** Still works but not recommended, will be removed in future
- **Removed:** No longer exists, manifests will fail

**Q: How long is the deprecation period?**
- **Stable APIs:** Minimum 12 months or 3 releases (whichever is longer)
- **Beta APIs:** Minimum 9 months or 3 releases

## Custom Resource Definitions (CRD)

### Overview
**CRDs extend Kubernetes API** by allowing you to define your own custom resource types, just like built-in resources (Pods, Deployments, etc.).

### Relationship

```
CRD (Template) → CR (Instance)
   ↓                ↓
Definition      Actual Resource

Example:
Fruit CRD → Apple CR, Orange CR, Banana CR
```

**Analogy:**
```
API Group    → Resource Type
─────────────────────────────
apps         → Deployment
batch        → Job, CronJob
mygroup      → Fruit (custom)
```

### Complete CRD Example

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: fruits.mygroup.example.com  # Must be: <plural>.<group>
spec:
  group: mygroup.example.com
  scope: Namespaced  # or Cluster
  names:
    plural: fruits       # kubectl get fruits
    singular: fruit      # kubectl get fruit apple
    kind: Fruit          # kubectl get Fruit
    shortNames:
    - fr                 # kubectl get fr
  versions:
  - name: v1
    served: true         # Accept requests for this version
    storage: true        # Store in etcd in this version
    schema:
      openAPIV3Schema:   # Defines structure using OpenAPI v3
        type: object
        properties:
          spec:
            type: object
            properties:
              color:
                type: string
              taste:
                type: string
              weight:
                type: number
            required: ["color"]  # color is mandatory
```

### Field Explanations

| Field | Purpose | Example |
|-------|---------|---------|
| `metadata.name` | Fully qualified CRD name | `fruits.mygroup.example.com` |
| `spec.group` | API group for the resource | `mygroup.example.com` |
| `spec.scope` | Namespace-scoped or Cluster-wide | `Namespaced` or `Cluster` |
| `names.plural` | Plural form | `fruits` |
| `names.singular` | Singular form | `fruit` |
| `names.kind` | Kubernetes Kind | `Fruit` |
| `names.shortNames` | Aliases | `fr` |
| `versions[].served` | Accept API requests | `true` |
| `versions[].storage` | Store in etcd | `true` |
| `schema` | Resource structure definition | OpenAPI v3 Schema |

### Advanced CRD Features

#### Validation

```yaml
schema:
  openAPIV3Schema:
    type: object
    properties:
      spec:
        type: object
        properties:
          color:
            type: string
            enum: ["red", "green", "yellow"]  # Only these values allowed
          weight:
            type: number
            minimum: 0.1
            maximum: 5.0

```

```yaml

  served: true

      type: object

        spec:
```

### CRD with Operator Pattern

**Operator = Custom Controller + CRD**

```
CRD (What you want) → Custom Controller (How to achieve it)
                                ↓
                    Manages native Kubernetes resources
```

**Example:** MySQL Operator
- CRD: `MySQL` resource
- Controller: Watches `MySQL` resources, creates Deployments, Services, ConfigMaps

### Interview Questions

**Q: What's the difference between CRD and ConfigMap?**
- **ConfigMap:** Stores configuration data
- **CRD:** Extends Kubernetes API with new resource types

**Q: When should you use CRDs?**
- Building Kubernetes operators
- Abstracting complex deployments
- Creating platform-specific resources
- Implementing custom controllers

**Q: What happens when you delete a CRD?**
- All Custom Resources (CRs) of that type are **deleted**
- Use finalizers to prevent accidental deletion

**Q: Can you have multiple versions of a CRD?**
- Yes! Support migration paths (v1alpha1 → v1beta1 → v1)
- Only ONE version can be `storage: true`

### Sample CKAD Questions

#### Question 1: PV, PVC, and Pod

**Task:** In namespace `mct`, create:
1. PersistentVolume `mct-pv`
   - Capacity: 100Mi
   - Access: ReadWriteOnce
   - HostPath: `/tmp/data`
   
2. PersistentVolumeClaim `mct-pvc`
   - Request: 100Mi
   - Access: ReadWriteOnce
   
3. Pod `nginx-mct`
   - Image: nginx
   - Container: mct-container
   - Mount: `/tmp/projectdata`

**Solution:**

```yaml
# PV
apiVersion: v1
kind: PersistentVolume
metadata:
  name: mct-pv
spec:
  capacity:
    storage: 100Mi
  accessModes:
  - ReadWriteOnce
  hostPath:
    path: "/tmp/data"

---
# PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mct-pvc
  namespace: mct
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 100Mi

---
# Pod
apiVersion: v1
kind: Pod
metadata:
  name: nginx-mct
  namespace: mct
spec:
  containers:
  - name: mct-container
    image: nginx
    volumeMounts:
    - mountPath: "/tmp/projectdata"
      name: mct-storage
  volumes:
  - name: mct-storage
    persistentVolumeClaim:
      claimName: mct-pvc
```

#### Question 2: CronJob

**Task:** Create CronJob `hello` that:
- Runs every minute
- Uses busybox image
- Executes `date` command
- Completes within 22 seconds or terminates

**Solution:**

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello
spec:
  schedule: "* * * * *"
  jobTemplate:
    spec:
      activeDeadlineSeconds: 22
      template:
        spec:
          containers:
          - name: hello
            image: busybox
            command: ["date"]
          restartPolicy: Never
```
