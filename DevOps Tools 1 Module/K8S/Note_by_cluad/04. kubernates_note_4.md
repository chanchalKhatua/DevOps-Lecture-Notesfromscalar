## StatefulSets

### Overview
A **StatefulSet** is a Kubernetes workload API object designed to manage **stateful applications** that require:
- Stable, unique network identities
- Stable, persistent storage
- Ordered, graceful deployment and scaling
- Ordered, automated rolling updates

### StatefulSet vs Deployment

| Feature | Deployment | StatefulSet |
|---------|-----------|-------------|
| **Use Case** | Stateless applications | Stateful applications (databases, etc.) |
| **Pod Identity** | Random names, no stable identity | Stable, unique identities (predictable names) |
| **Storage** | Ephemeral or shared storage | Persistent storage with PVCs |
| **Scaling** | Pods scaled randomly | Pods scaled sequentially (0→1→2) |
| **Service Type** | Normal Service (load balancing) | Headless Service (unique DNS per pod) |
| **Pod Naming** | `app-xyz-12345678` | `app-xyz-0`, `app-xyz-1`, `app-xyz-2` |

```yaml
#Pods scaled sequentially (0→1→2) This is true by default, BUT incomplete.
#Add: podManagementPolicy: Parallel
#Explanation: OrderedReady = default
#Parallel = pods created simultaneously

apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: fast-stateful-workers
spec:
  serviceName: "worker-svc"
  replicas: 5
  podManagementPolicy: Parallel # <--- Overrides default OrderedReady behavior
  selector:
    matchLabels:
      app: batch-worker
  template:
    metadata:
      labels:
        app: batch-worker
    spec:
      containers:
      - name: worker
        image: my-worker-image:v1.2
  ```

### Key Concepts

#### Pod Naming Convention
```
<statefulset-name>-<ordinal-index>
Example: my-db-0, my-db-1, my-db-2
```

#### Headless Service
- Created with `clusterIP: None`
- Provides unique DNS entries for each pod
- DNS format: `<pod-name>.<service-name>.<namespace>.svc.cluster.local`
- Example: `my-db-0.db-service.my-namespace.svc.cluster.local`

### Complete StatefulSet Example

```yaml
# Headless Service
apiVersion: v1
kind: Service
metadata:
  name: db-service
spec:
  clusterIP: None  # Makes service headless
  selector:
    app: my-db
  ports:
  - port: 5432
    targetPort: 5432

---
# StatefulSet
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: my-db
spec:
  serviceName: db-service  # Links to headless service
  replicas: 3
  selector:
    matchLabels:
      app: my-db
  template:
    metadata:
      labels:
        app: my-db
    spec:
      containers:
      - name: database
        image: postgres
        env:
        - name: POSTGRES_PASSWORD
          value: "mysecretpassword"
        ports:
        - containerPort: 5432
        volumeMounts:
        - name: db-storage
          mountPath: /var/lib/postgresql/data
  
  # Volume Claim Templates - creates separate PVC for each pod
  volumeClaimTemplates:
  - metadata:
      name: db-storage
    spec:
      accessModes: ["ReadWriteOnce"]
      resources:
        requests:
          storage: 5Gi
```

### Interview Questions
**Q: Why use StatefulSets instead of Deployments for databases?**
- Stable network identity (each pod has predictable DNS name)
- Persistent storage that survives pod rescheduling
- Ordered scaling ensures data consistency
- Individual pod addressing for master-slave configurations

**Q: What happens when a StatefulSet pod is deleted?**
- Pod is recreated with the **same name and identity**
- Reattaches to the **same PVC** (data persists)
- Maintains the same ordinal index

## Kubernetes Security Framework

### Security Layers
```
┌─────────────────────────────────────┐
│   1. Authentication (Who are you?)  │
├─────────────────────────────────────┤
│   2. Authorization (What can you do?)│
├─────────────────────────────────────┤
│   3. Admission Control (Policy checks)│
└─────────────────────────────────────┘
```

### API Request Flow
```
User/Service → Authentication → Authorization → Admission Controllers → API Server → etcd
```

## Authentication Methods

### Overview
**Authentication verifies the identity** of users and systems interacting with the Kubernetes cluster.

### 1. Client Certificates (X.509)

**How it works:**
- Uses X.509 certificates to authenticate users
- Certificate issued by Kubernetes CA
- Strong cryptographic validation

**Use Case:** Secure authentication for admins and system processes

**Pros:**
- Strong security
- Cryptographic validation

**Cons:**
- Requires certificate management
- Complex setup

**Workflow:**
```
1. Generate private key: openssl genrsa -out user.key 2048
2. Create CSR: openssl req -new -key user.key -out user.csr
3. Submit CSR to Kubernetes CA
4. CA signs and returns certificate
5. Use certificate in kubeconfig
```

### 2. Service Account Tokens (Bearer Token)

**How it works:**
- Kubernetes automatically generates tokens for pods
- Token mounted at `/var/run/secrets/kubernetes.io/serviceaccount/token`
- Pods use this token to authenticate with API server

**Use Case:** Applications running inside the cluster

**Token Characteristics:**
- Automatically mounted to pods
- Scoped to namespace
- Can be rotated for security

**Pros:**
- Automatic, easy to use
- Built into Kubernetes

**Cons:**
- Security risks if token leaked
- Requires token rotation strategy

**Example:**
```bash
# View token inside pod
kubectl exec -it <pod-name> -- cat /var/run/secrets/kubernetes.io/serviceaccount/token
```

### 3. Static Token File

**How it works:**
- Predefined tokens stored in CSV file
- Format: `token,username,uid,group`
- Configured in API server manifest

**Use Case:** Testing or simple setups (NOT production)

**Configuration:**
```csv
# /etc/kubernetes/static_tokens.csv
my-static-token,user1,user1,system:masters
```

API Server configuration:
```yaml
--token-auth-file=/etc/kubernetes/static_tokens.csv
```

**Pros:**
- Easy setup

**Cons:**
- **INSECURE** - token never expires
- Requires API server restart to update
- Manual management

### 4. OpenID Connect (OIDC) Authentication

**How it works:**
- Integrates with external identity providers (Google, Azure, Okta)
- Uses OAuth 2.0/OIDC protocol
- Centralized identity management

**Flow:**
```
User → Identity Provider → Token → Kubernetes API Server
```

**Use Case:** Enterprise environments with centralized authentication

**Pros:**
- Centralized identity management
- Easy integration with corporate credentials
- Supports MFA

**Cons:**
- Requires external identity provider
- More complex setup

**Example Identity Providers:**
- Google
- Azure Active Directory
- Okta
- Keycloak

### 5. Webhook Token Authentication

**How it works:**
- API server calls external webhook to validate tokens
- Custom authentication logic
- Flexible authentication service

**Flow:**
```
User Request → API Server → Webhook Server → Validation Response → Access Decision
```

**Use Case:** Custom authentication systems

**Configuration:**
```yaml
# API Server flags
--authentication-token-webhook-config-file=/etc/kubernetes/webhook-config.yaml
--authentication-token-webhook-cache-ttl=5m
```

**Webhook Config:**
```yaml
apiVersion: v1
kind: Config
clusters:
- name: webhook-auth
  cluster:
    server: https://auth.example.com/validate
users:
- name: webhook-auth
contexts:
- context:
    cluster: webhook-auth
    user: webhook-auth
  name: webhook-auth
current-context: webhook-auth
```

**Pros:**
- Flexible, integrates with external systems
- Custom authentication logic

**Cons:**
- Requires maintaining separate authentication service
- Additional complexity

### Authentication Methods Comparison

| Method | Security | Complexity | Use Case | Recommended |
|--------|----------|----------|----------|-------------|
| **Client Certificates** | High | Medium | Admin access | ✅ Yes |
| **Service Account Tokens** | Medium | Low | In-cluster apps | ✅ Yes |
| **Static Token File** | Low | Low | Testing only | ❌ No (production) |
| **OIDC** | High | High | Enterprise SSO | ✅ Yes |
| **Webhook** | Medium-High | High | Custom systems | ⚠️ Case-by-case |

## Authorization Modes

### Overview
**Authorization determines what actions an authenticated user can perform** within the cluster.

### 1. Node Authorization

**Purpose:** Authorizes kubelets to read and write node-specific objects

**How it works:**
- Used by kubelet on each node
- Limited to node-specific operations
- Automatic, built-in

**Permissions:**
- Read: pods, services, endpoints
- Write: node status, pod status, events

### 2. ABAC (Attribute-Based Access Control)

**How it works:**
- Policies stored in static JSON file
- Each policy defines user, resource, and actions
- Requires API server restart for updates

**Policy Example:**
```json
{
  "apiVersion": "abac.authorization.kubernetes.io/v1beta1",
  "kind": "Policy",
  "spec": {
    "user": "alice",
    "namespace": "default",
    "resource": "pods",
    "readonly": true
  }
}
```

**API Server Configuration:**
```bash
--authorization-mode=Node,RBAC,ABAC
--authorization-policy-file=/etc/kubernetes/abac-policy.json
```

**Why ABAC is Less Preferred:**
- ❌ Static file-based (requires API server restart)
- ❌ Not scalable for large clusters
- ❌ Difficult to manage
- ✅ RBAC is preferred (dynamic, easier to manage)

### 3. Webhook Authorization

**How it works:**
- Delegates authorization decisions to external service
- API server sends request to webhook
- Webhook validates and returns allow/deny

**Flow:**
```
API Request → API Server → Webhook Server → Authorization Decision → Allow/Deny
```

**Configuration:**
```yaml
apiVersion: v1
kind: Config
clusters:
- name: webhook-auth
  cluster:
    server: https://auth.example.com/validate
users:
- name: webhook-auth
contexts:
- context:
    cluster: webhook-auth
    user: webhook-auth
  name: webhook-auth
current-context: webhook-auth
```

### 4. RBAC (Role-Based Access Control) ⭐ **RECOMMENDED**

#### Key Concepts

**Roles:** Define permissions (what can be done)
**RoleBindings:** Assign roles to users/groups (who can do it)

#### Components

1. **Role** - Namespace-scoped permissions
2. **ClusterRole** - Cluster-wide permissions
3. **RoleBinding** - Assigns Role to subjects within namespace
4. **ClusterRoleBinding** - Assigns ClusterRole to subjects cluster-wide

#### Role Example
```yaml
kind: Role
apiVersion: rbac.authorization.k8s.io/v1
metadata:
  namespace: default
  name: pod-reader
rules:
- apiGroups: [""]  # "" indicates core API group
  resources: ["pods"]
  verbs: ["get", "watch", "list"]
```

#### RoleBinding Example
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: RoleBinding
metadata:
  name: read-pods
  namespace: default
subjects:
- kind: User
  name: "jane"
  apiGroup: rbac.authorization.k8s.io
- kind: Group
  name: "dev-team"
  apiGroup: rbac.authorization.k8s.io
- kind: ServiceAccount
  name: "app-sa"
  namespace: default
roleRef:
  kind: Role
  name: pod-reader
  apiGroup: rbac.authorization.k8s.io
```

#### ClusterRole Example (Cluster-wide)
```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: cluster-admin-reader
rules:
- apiGroups: ["*"]
  resources: ["*"]
  verbs: ["get", "list", "watch"]
```

### RBAC API Groups & Resources

| API Group | Example Resources |
|-----------|-------------------|
| `""` (core) | pods, services, configmaps, secrets, nodes, namespaces |
| `apps` | deployments, daemonsets, statefulsets, replicasets |
| `batch` | jobs, cronjobs |
| `rbac.authorization.k8s.io` | roles, rolebindings, clusterroles, clusterrolebindings |
| `networking.k8s.io` | networkpolicies, ingresses |
| `storage.k8s.io` | storageclasses, volumeattachments |

### RBAC Verbs (Actions)

| Verb | Description |
|------|-------------|
| `get` | Read a specific resource |
| `list` | List all resources of a type |
| `watch` | Monitor changes to resources in real-time |
| `create` | Create a new resource |
| `update` | Modify an existing resource |
| `patch` | Partially update an existing resource |
| `delete` | Remove a resource |
| `deletecollection` | Delete multiple resources at once |

### Authorization Demo Workflow

```
1. Generate key → openssl genrsa -out dev-user.key 2048
2. Create CSR → openssl req -new -key dev-user.key -out dev-user.csr
3. Submit CSR to CA → kubectl certificate approve
4. Create Role → kubectl create -f role.yaml
5. Create RoleBinding → kubectl create -f rolebinding.yaml
6. Configure kubeconfig with certificate and key
7. Test permissions → kubectl auth can-i get pods --as dev-user
```

## Security Contexts

### Overview

Security contexts define privilege and access control settings for pods and containers.

### Two Levels of Application

1. **Pod-level:** Applies globally to all containers within the pod.
2. **Container-level:** Applies only to a specific container (overrides pod-level settings if there is a conflict).

### Pod-Level Security Context

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:       # Applies to ALL containers
    runAsUser: 1000      # Run as non-root user (UID 1000)
    runAsGroup: 3000     # Group ID
    fsGroup: 2000        # File system group ownership
  containers:
  - name: app-container
    image: nginx

```

### Container-Level Security Context

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-container
spec:
  containers:
  - name: app-container
    image: nginx
    securityContext:     # Only applies to THIS container
      runAsUser: 2000
      readOnlyRootFilesystem: true
      allowPrivilegeEscalation: false

```

---

### Key Security Context Fields

**1. `runAsUser` & `runAsGroup**`

```yaml
securityContext:
  runAsUser: 1000   # Runs as non-root user (UID 1000)
  runAsGroup: 3000  # Uses GID 3000

```

> **Interview Question:**
> **Q:** Why run as a non-root user?
> **A:** It follows the principle of least privilege, limits potential damage from a container compromise, and prevents privilege escalation to the underlying host.

**2. Privileged Mode**

```yaml
securityContext:
  privileged: true  # Grants FULL access to host

```

* **⚠️ Dangerous:** The container has unrestricted access to host resources. Avoid unless building system-level components.

**3. `allowPrivilegeEscalation**`

```yaml
securityContext:
  allowPrivilegeEscalation: false  # Prevents gaining more privileges

```

* **Recommended:** Always set to `false` unless absolutely necessary.

**4. `readOnlyRootFilesystem**`

```yaml
securityContext:
  readOnlyRootFilesystem: true  # Prevents modifying root filesystem

```

* **Benefits:** Prevents malware installation, ensures immutable containers, and heavily improves your security posture.

**5. Linux Capabilities**

```yaml
securityContext:
  capabilities:
    add: ["NET_ADMIN"]   # Allows network configuration
    drop: ["ALL"]        # Drops all unnecessary capabilities

```

* **Common Capabilities:** `NET_ADMIN` (Network config), `SYS_TIME` (Set system time), `CHOWN` (Change file ownership).

---

### Complete Security Context Example

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: secure-pod
spec:
  securityContext:  # Pod-level
    runAsUser: 1000
    runAsGroup: 3000
    fsGroup: 2000
  containers:
  - name: secure-container
    image: busybox
    securityContext:  # Container-level
      allowPrivilegeEscalation: false
      readOnlyRootFilesystem: true
      capabilities:
        drop: ["ALL"]
    command: ["sleep", "3600"]

```

### Testing Security Contexts

```bash
# Check user ID
kubectl exec -it secure-pod -- id

# Try to escalate to root (should fail)
kubectl exec -it secure-pod -- sh -c "su root"

# Try to create file in root (should fail with readOnlyRootFilesystem)
kubectl exec -it secure-pod -- touch /testfile

```

###  ServiceAccount Token Hardening

*Note: This is a senior-level security concept, typically discussed alongside RBAC or Security Contexts.*

**Overview**
By default, Kubernetes automatically mounts ServiceAccount tokens into every pod.

**Security Risk**
If a pod is compromised, an attacker can easily extract the automatically mounted token and use it to authenticate against the Kubernetes API, potentially escalating their privileges across the cluster.

**Best Practice**
Disable automatic token mounting globally on the Pod (or directly on the ServiceAccount) unless the application specifically requires Kubernetes API access.

```yaml
spec:
  automountServiceAccountToken: false

```

> **Interview Tip:**
> If an interviewer asks how to secure ServiceAccounts, mention `automountServiceAccountToken: false` first. Then, add that if a pod *does* need API access, you should create a dedicated ServiceAccount for it with the absolute minimum RBAC permissions, rather than relying on the `default` ServiceAccount.

### Runtime Security

**Overview**
While RBAC and Pod Security Admission control *how* resources are deployed, runtime security focuses on what containers are actually allowed to do at the Linux kernel level once they are running. This is a highly tested domain for the CKS (Certified Kubernetes Security Specialist) exam and advanced technical interviews.

#### 1. seccomp (Secure Computing Mode)

* **Why you need it:** By default, Linux containers have access to hundreds of system calls (syscalls). Most applications only need a fraction of these. Seccomp restricts which syscalls a container can make, severely limiting the attack surface if an attacker breaks into your container and tries to execute kernel exploits.
* **How it works:** You define a seccomp profile natively in the pod's `securityContext`. Setting it to `RuntimeDefault` automatically drops dangerous syscalls while allowing standard operations.
* **Example:**
```yaml
securityContext:
  seccompProfile:
    type: RuntimeDefault

```

#### 2. AppArmor (Application Armor)
*   **Why you need it:** AppArmor is a Mandatory Access Control (MAC) system that restricts a container's access to files, network ports, and POSIX capabilities. If an attacker gains root access inside a container, a strict AppArmor profile can still stop them from modifying critical files or executing malicious binaries.
*   **How it works:** The AppArmor profile must first be loaded directly onto the underlying worker nodes. You then instruct the pod to use that specific profile.
*   **Example:** *(Historically applied via annotations, but native in v1.30+)*
    ```yaml
    securityContext:
      appArmorProfile:
        type: Localhost
        localhostProfile: "k8s-apparmor-custom-profile"
    ```

#### 3. SELinux (Security-Enhanced Linux)

* **Why you need it:** Standard on RHEL-based systems, SELinux is another MAC system. Instead of focusing heavily on paths like AppArmor, it uses strict label-based security, applying contexts (user, role, type, level) to processes and files to dictate exactly who can access what.
* **How it works:** Applied via the `seLinuxOptions` block inside the `securityContext`. The underlying Linux node must be running with SELinux enabled for this to take effect.
* **Example:**
```yaml
securityContext:
  seLinuxOptions:
    level: "s0:c123,c456"

```

#### 4. Falco (Runtime Threat Detection)
*   **Why you need it:** Unlike the first three tools which *prevent* actions, Falco *detects* them. Created by Sysdig and now an incubated CNCF project, Falco acts like a security camera for your cluster. It alerts you to anomalous behavior in real-time (e.g., "A shell was unexpectedly spawned inside a container" or "A sensitive file was modified").
*   **How it works:** Usually deployed as a DaemonSet across the cluster. It hooks directly into the Linux kernel (typically via eBPF) to monitor all syscalls, compares them against a predefined rules engine, and forwards alerts to logs, SIEMs, or Slack.

> **Interview Tip:**
> If an interviewer asks you how to handle "Runtime Security," categorize the tools clearly to show you understand the full landscape:
> *   **Prevention (Kernel-level restrictions):** seccomp, AppArmor, SELinux.
> *   **Detection (Monitoring & Alerting):** Falco.


### Security Context Best Practices

**✅ DO:**

* Always run as a non-root user.
* Set `allowPrivilegeEscalation: false`.
* Use `readOnlyRootFilesystem: true`.
* Drop all unnecessary capabilities (`drop: ["ALL"]`).
* Use `fsGroup` for shared volume access.

**❌ DON'T:**

* Run containers as root (UID 0).
* Use `privileged: true` unless absolutely necessary (e.g., CNI plugins).
* Allow privilege escalation.
* Grant unnecessary Linux capabilities.

## Kubernetes Secrets Security

### The Important Reality (Common Interview Trap)

A very common misconception is that Kubernetes Secrets are inherently secure.

* **Kubernetes Secrets are merely Base64 ENCODED, they are NOT ENCRYPTED by default.**
* Base64 is a data encoding scheme, not encryption. Anyone can decode a Base64 string in seconds using `echo "cGFzc3dvcmQ=" | base64 --decode`.

### The Security Risks

Because secrets are not encrypted by default, they pose significant risks:

1. **etcd Access:** All Kubernetes objects, including Secrets, are stored in the `etcd` database. Anyone with direct access to the `etcd` backend (or the nodes running it) can read all cluster secrets in plain text.
2. **RBAC Misconfigurations:** If a user or service account is overly provisioned with `get/list` permissions on `secrets` within a namespace, they can easily retrieve and decode sensitive data.
3. **GitOps Risks:** Hardcoding plain-text or Base64 secrets into YAML files and pushing them to version control (like GitHub) exposes credentials to the entire engineering team.

---

### Solutions for Securing Secrets

To properly secure secrets in a production environment, you must implement one (or both) of the following strategies:

#### 1. Enable Native Encryption at Rest

You can configure the Kubernetes API server to encrypt secret data *before* it is written to the `etcd` database.

* **How it works:** You pass an encryption configuration file to the `kube-apiserver` startup arguments.
* **The Flag:**
```bash
--encryption-provider-config=/etc/kubernetes/encryption-config.yaml

```


* *Note: This protects the data inside `etcd`, but users with `kubectl get secret` permissions will still see the decoded values.*

#### 2. Use External Secret Managers (Enterprise Standard)

Instead of relying on Kubernetes to act as a vault, modern production environments store secrets in a dedicated external Secret Management system.

* **HashiCorp Vault** (The industry standard for multi-cloud)
* **AWS Secrets Manager**
* **Azure Key Vault**
* **GCP Secret Manager**

---

### Best Practices for External Integration

When using an external secret manager, how do you get the secrets into the Pods?

**1. Secrets Store CSI Driver (Recommended)**
This allows Kubernetes to mount multiple secrets, keys, and certs stored in enterprise external secret managers into pods as a volume. Once the volume is attached, the data is available to the container file system.

**2. External Secrets Operator (ESO)**
An alternative popular approach that reads information from external APIs (like AWS Secrets Manager) and automatically synchronizes the values into native Kubernetes Secret objects.

---

> **Interview Question:**
> **Q:** If I create a Kubernetes Secret, is it encrypted in the database? How would you secure it?
> **A:** No, by default Kubernetes Secrets are only Base64 encoded, meaning anyone with access to `etcd` can read them. To secure them, we need to enable native Encryption at Rest via the API server's `--encryption-provider-config` flag. However, the production best practice is to offload secrets entirely to an external tool like HashiCorp Vault or AWS Secrets Manager and inject them into pods using the Secrets Store CSI Driver.

## Pod Security Admission (PSA)

### Overview

Pod Security Admission (PSA) is the native Kubernetes mechanism that enforces security standards on pods at the **namespace level**. It replaced PodSecurityPolicy (PSP), which was completely removed in Kubernetes **v1.25**.

### Levels vs. Modes

PSA operates by applying a security **Level** using a specific operational **Mode**.

**1. Security Levels (What is restricted):**

| Level | Purpose |
| --- | --- |
| **Privileged** | Unrestricted policy; allows known privilege escalations. Used for system-level or infrastructure workloads. |
| **Baseline** | Minimally restrictive policy; prevents known privilege escalations but allows default pod configurations. |
| **Restricted** | Heavily restricted policy; enforces strict pod hardening best practices. Recommended for production. |

**2. Operational Modes (How it is applied):**

* **Enforce:** Policy violations will cause the pod to be rejected.
* **Audit:** Violations are allowed but recorded in the audit log.
* **Warn:** Violations are allowed but trigger a user-facing warning.

### Example: Labeling a Namespace

You enforce these policies by applying labels to a namespace. The label syntax is always `pod-security.kubernetes.io/<MODE>=<LEVEL>`.

```bash
# Enforce the restricted policy on the 'production' namespace
kubectl label ns production pod-security.kubernetes.io/enforce=restricted

# You can also apply multiple modes at once (e.g., enforce baseline, but warn on restricted)
kubectl label ns production \
  pod-security.kubernetes.io/enforce=baseline \
  pod-security.kubernetes.io/warn=restricted

```

### Common "Restricted" Policy Requirements

If a namespace enforces the **Restricted** level, pods will be rejected unless they explicitly configure their `securityContext` to ensure:

* **Non-root containers:** `runAsNonRoot: true`
* **No privilege escalation:** `allowPrivilegeEscalation: false`
* **No privileged containers:** `privileged: false`
* **Drop dangerous capabilities:** `capabilities: { drop: ["ALL"] }`
* **No host namespaces/paths:** Cannot mount `hostPath` volumes or share the host's network/PID namespaces.

> **Interview Question:**
> **Q:** What replaced PodSecurityPolicy, and how is it applied?
> **A:** Pod Security Admission (PSA) replaced PSP in Kubernetes v1.25. Unlike PSP which was applied cluster-wide via roles, PSA is enforced at the **namespace level** using specific namespace labels.

## Admission Controllers

### Overview
**Admission controllers intercept API requests AFTER authentication and authorization** but BEFORE the object is persisted to etcd.

### Request Flow
```
API Request → Authentication → Authorization → Admission Controllers → etcd
                                                      ↓
                                            Validate/Modify/Reject
```

### Types of Admission Controllers

#### 1. Mutating Admission Controllers
- **Modify** objects before they are stored
- Examples: Add default labels, inject sidecars, set resource limits

#### 2. Validating Admission Controllers
- **Validate** objects against policies
- **Cannot modify** objects
- Examples: Enforce resource quotas, reject insecure configurations

#### 3. Dual-Purpose Controllers
- Both mutate AND validate
- Example: **LimitRanger**

### Execution Order

## Finalizers in Kubernetes

### Overview & Purpose

Finalizers are namespace-scoped keys that tell Kubernetes to wait until specific conditions are met before it fully deletes a resource. They act as pre-delete hooks, ensuring that asynchronous cleanup operations complete successfully, preventing accidental deletion and orphaned resources.

### The Deletion Workflow

When you delete an object with a finalizer, Kubernetes does not immediately remove it from `etcd`. Instead, it updates the object:

1. **Delete Request:** User issues `kubectl delete`.
2. **Timestamp Added:** Kubernetes adds a `deletionTimestamp` to the resource.
3. **Terminating State:** The resource enters the `Terminating` state.
4. **Controller Cleanup:** The respective controller sees the timestamp, executes its cleanup logic, and removes its finalizer from the list.
5. **Resource Deleted:** Once the `finalizers` list is empty, Kubernetes removes the object from `etcd`.

### Example

```yaml
metadata:
  name: my-pvc
  finalizers:
  - kubernetes.io/pvc-protection

```

### Common Real-World Cases

* **PVC Protection:** `kubernetes.io/pvc-protection` ensures a PersistentVolumeClaim is not deleted while a Pod is still actively using it.
* **Load Balancer Cleanup:** Ensures cloud-provider LoadBalancers (AWS ELB, Azure LB) are actually destroyed in the cloud before the Kubernetes Service is removed.
* **Namespace Deletion:** Ensures all resources *inside* a namespace are deleted before the namespace itself is removed.

### ⚠️ Troubleshooting: "Stuck in Terminating"

> **Interview Question:** > **Q:** A namespace or pod is stuck in the "Terminating" state for a long time. What is the likely cause and how do you fix it?
> **A:** This is almost always caused by a stuck finalizer waiting for a cleanup process that failed or no longer exists. You can resolve it by editing the resource (`kubectl edit <resource>`) and manually removing the finalizer from the `metadata.finalizers` array, or by using a `kubectl patch` command to nullify it:
> ```bash
> kubectl patch pvc my-pvc -p '{"metadata":{"finalizers":null}}'
> 
> ```

## Gateway API

### Overview

The Gateway API is the next-generation, official Kubernetes replacement for the traditional `Ingress` resource. While Ingress is still widely used, Gateway API addresses its limitations by providing a richer, more expressive, and extensible standard for service networking.

### Key Benefits over Ingress

* **More Expressive Routing:** Natively supports header-based matching, traffic weighting (canary deployments), and multi-protocol routing (HTTP, gRPC, TCP, UDP) without relying on custom annotations.
* **Role-Oriented Architecture:** Designed to be used by different personas (Infrastructure Providers, Cluster Operators, and App Developers).
* **Better Multi-Team Separation:** Developers can manage their own application routing rules independently of the cluster's core load-balancing infrastructure.

### Core Components

| Resource | Persona | Purpose |
| --- | --- | --- |
| **GatewayClass** | Infrastructure Provider | Defines the infrastructure template (e.g., AWS ALB, NGINX). |
| **Gateway** | Cluster Operator | Instantiates the load balancer entry point and defines ports/listeners. |
| **HTTPRoute** | App Developer | Defines the actual routing rules (paths, headers, traffic splitting) to the backend services. |

> **Interview Tip:** If asked about routing external traffic, mention standard Ingress first, but score bonus points by stating: *"Ingress is still the standard today, but the Gateway API is the future direction of Kubernetes networking due to its role-based model and native CRD support."*

## Native Sidecar Containers (v1.29+)

### Overview

Historically, Kubernetes did not have a native concept of a "sidecar" container; developers simply added a second regular container to a pod. This caused issues with container startup order and prevented Kubernetes Jobs from completing (because the sidecar would run forever).

As of Kubernetes **v1.29**, sidecars are natively supported by configuring them as **restartable init containers**.

### Example

To create a native sidecar, you place it under `initContainers` and set the `restartPolicy` to `Always`.

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: web-app-with-sidecar
spec:
  initContainers:
  - name: log-agent-sidecar
    image: busybox
    restartPolicy: Always   # <--- This makes it a native sidecar!
    command: ["sh", "-c", "tail -f /var/log/app.log"]
  containers:
  - name: main-app
    image: nginx

```

### Key Benefits

1. **Guaranteed Startup Order:** Unlike regular containers, init containers start sequentially. A native sidecar will start and report "Ready" *before* the main application container begins booting.
2. **Graceful Job Completion:** If used in a Kubernetes `Job`, native sidecars will automatically terminate when the main application container finishes its task, preventing the Job from hanging indefinitely.
3. **Continuous Running:** Because of `restartPolicy: Always`, it continues to run alongside the main container for the entire lifecycle of the Pod.

## Custom Resource Definitions (CRD)

### Overview
**CRDs extend Kubernetes API** by allowing you to define your own custom resource types, just like built-in resources (Pods, Deployments, etc.).

### Relationship

```
CRD (Template) → CR (Instance)
   ↓                ↓
Definition      Actual Resource

Example:
Fruit CRD → Apple CR, Orange CR, Banana CR
```

**Analogy:**
```
API Group    → Resource Type
─────────────────────────────
apps         → Deployment
batch        → Job, CronJob
mygroup      → Fruit (custom)
```

### Complete CRD Example

```yaml
apiVersion: apiextensions.k8s.io/v1
kind: CustomResourceDefinition
metadata:
  name: fruits.mygroup.example.com  # Must be: <plural>.<group>
spec:
  group: mygroup.example.com
  scope: Namespaced  # or Cluster
  names:
    plural: fruits       # kubectl get fruits
    singular: fruit      # kubectl get fruit apple
    kind: Fruit          # kubectl get Fruit
    shortNames:
    - fr                 # kubectl get fr
  versions:
  - name: v1
    served: true         # Accept requests for this version
    storage: true        # Store in etcd in this version
    schema:
      openAPIV3Schema:   # Defines structure using OpenAPI v3
        type: object
        properties:
          spec:
            type: object
            properties:
              color:
                type: string
              taste:
                type: string
              weight:
                type: number
            required: ["color"]  # color is mandatory
```

### Field Explanations

| Field | Purpose | Example |
|-------|---------|---------|
| `metadata.name` | Fully qualified CRD name | `fruits.mygroup.example.com` |
| `spec.group` | API group for the resource | `mygroup.example.com` |
| `spec.scope` | Namespace-scoped or Cluster-wide | `Namespaced` or `Cluster` |
| `names.plural` | Plural form | `fruits` |
| `names.singular` | Singular form | `fruit` |
| `names.kind` | Kubernetes Kind | `Fruit` |
| `names.shortNames` | Aliases | `fr` |
| `versions[].served` | Accept API requests | `true` |
| `versions[].storage` | Store in etcd | `true` |
| `schema` | Resource structure definition | OpenAPI v3 Schema |

```yaml

  namespace: default
```

## Common Kubernetes Interview Traps

#### 1. Service Discovery & Selectors

**The Trap:** Candidates often assume a Service routes traffic to a Deployment by referencing the Deployment's name.
**The Reality:** Services have absolutely no knowledge of Deployments or ReplicaSets. They route traffic purely based on **Labels**.

* **Why:** This decoupled architecture allows a single Service to route traffic to Pods managed by different controllers, or to standalone Pods, as long as the labels match.
* **How:** The `spec.selector` in the Service YAML must exactly match the `metadata.labels` defined in the Pod template of your Deployment. If you change a Pod's label at runtime, it immediately falls out of the Service's load balancer pool.

#### 2. Network Policies and CNI Plugins

**The Trap:** Writing a perfect `NetworkPolicy` YAML, applying it, and assuming the cluster is secure.
**The Reality:** Kubernetes does not enforce Network Policies; it only stores them. They require a supporting Container Network Interface (CNI) plugin to actually enforce the packet filtering.

* **Why:** Kubernetes handles orchestration, delegating networking to external plugins.
* **How:** If your cluster uses **Flannel**, your Network Policies will be successfully created in the API but completely ignored in reality. You must use a CNI that supports enforcement, such as **Calico**, **Cilium**, or **Weave Net**.

#### 3. StatefulSets and PVC Deletion

**The Trap:** Assuming that deleting a StatefulSet cleans up everything it created, just like a Deployment.
**The Reality:** Deleting a StatefulSet (or scaling it down) **does NOT automatically delete its associated PersistentVolumeClaims (PVCs).**

* **Why:** StatefulSets are designed for stateful applications (like databases). Kubernetes prioritizes data safety; it assumes you might want to recover the data later or reattach it to a new StatefulSet.
* **How:** To clean up, you must manually delete the PVCs. *(Bonus Interview Points: Mention that starting in v1.27, you can configure the `persistentVolumeClaimRetentionPolicy` to auto-delete PVCs on StatefulSet deletion, but the default behavior is still to retain them).*

#### 4. Security Context Inheritance

**The Trap:** Believing that a pod-level `securityContext` strictly enforces rules for all containers inside it without exception.
**The Reality:** Container-level security contexts always **override** pod-level security contexts if there is a conflict.

* **Why:** Granularity. You might want the overall pod to run as user `1000`, but a specific utility sidecar container needs to run as user `2000`.
* **How:** If `runAsUser` is set to `1000` at the pod level and `2000` at the container level, that specific container runs as `2000`. The rest of the pod runs as `1000`.

#### 5. RBAC: RoleBinding to a ClusterRole

**The Trap:** Thinking that a `ClusterRole` can only be bound with a `ClusterRoleBinding`, resulting in cluster-wide permissions.
**The Reality:** A **`RoleBinding`** (which is namespace-scoped) CAN reference a **`ClusterRole`** (which is cluster-scoped).

* **Why:** Reusability. Instead of creating a duplicate `Role` named "edit" in 50 different namespaces, Kubernetes provides default `ClusterRoles` (like `admin`, `edit`, `view`).
* **How:** When you use a `RoleBinding` to grant a `ClusterRole` to a user, that user gets those permissions **only within the specific namespace** of the `RoleBinding`, not cluster-wide.

#### 6. Admission Controllers Execution Order

**The Trap:** Confusing the order of operations when objects are submitted to the API server.
**The Reality:** **Mutating** admission webhooks always run BEFORE **Validating** admission webhooks.

* **Why:** Logic dictates that you cannot validate an object's final state if another process is still allowed to modify it.
* **How:** The exact API flow is: Authentication $\rightarrow$ Authorization (RBAC) $\rightarrow$ **Mutating Admission** (modifies the payload, e.g., injecting a sidecar) $\rightarrow$ Object Schema Validation $\rightarrow$ **Validating Admission** (checks if the final payload violates policies) $\rightarrow$ Persisted to `etcd`.

   - HostPath: `/tmp/data`
   
2. PersistentVolumeClaim `mct-pvc`
   - Request: 100Mi
   - Access: ReadWriteOnce
   
3. Pod `nginx-mct`
   - Image: nginx
   - Container: mct-container
   - Mount: `/tmp/projectdata`

**Solution:**

```yaml
# PV
apiVersion: v1
kind: PersistentVolume
metadata:
  name: mct-pv
spec:
  capacity:
    storage: 100Mi
  accessModes:
  - ReadWriteOnce
  hostPath:
    path: "/tmp/data"

---
# PVC
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mct-pvc
  namespace: mct
spec:
  accessModes:
  - ReadWriteOnce
  resources:
    requests:
      storage: 100Mi

---
# Pod
apiVersion: v1
kind: Pod
metadata:
  name: nginx-mct
  namespace: mct
spec:
  containers:
  - name: mct-container
    image: nginx
    volumeMounts:
    - mountPath: "/tmp/projectdata"
      name: mct-storage
  volumes:
  - name: mct-storage
    persistentVolumeClaim:
      claimName: mct-pvc
```

#### Question 2: CronJob

**Task:** Create CronJob `hello` that:
- Runs every minute
- Uses busybox image
- Executes `date` command
- Completes within 22 seconds or terminates

**Solution:**

```yaml
apiVersion: batch/v1
kind: CronJob
metadata:
  name: hello
spec:
  schedule: "* * * * *"
  jobTemplate:
    spec:
      activeDeadlineSeconds: 22
      template:
        spec:
          containers:
          - name: hello
            image: busybox
            command: ["date"]
          restartPolicy: Never
```
